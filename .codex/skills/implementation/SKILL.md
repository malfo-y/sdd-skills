# Implementation Execution (TDD + Built-in Review)

Execute an implementation plan with dependency-aware TDD, phase reviews, and a final review report.

## Hard Rules

- Do not create, edit, rename, or delete files under `_sdd/spec/`.
- If spec drift is discovered, record it in progress notes and hand off to `spec-update-todo` or `spec-update-done`.
- Before running tests or runtime commands, apply `_sdd/env.md` when present.

## Inputs

- Primary plan sources:
  - `_sdd/implementation/IMPLEMENTATION_PLAN.md`
  - `_sdd/implementation/IMPLEMENTATION_PLAN_PHASE_<n>.md` (if split)
- Feature-draft plan source:
  - `_sdd/drafts/feature_draft_<feature_name>.md` (use **Part 2: Implementation Plan**)
- Existing progress (if any): `_sdd/implementation/IMPLEMENTATION_PROGRESS*.md`
- Project test/code conventions from repository code.

If multiple plan sources exist and user did not specify:

- If only one source exists, use it.
- If both `IMPLEMENTATION_PLAN*` and feature draft(s) exist:
  - prefer `IMPLEMENTATION_PLAN*` when they target the same feature
  - otherwise ask user which feature/plan to execute
- If only multiple feature drafts exist, ask user which draft to execute.

## Output

- Progress artifacts:
  - `_sdd/implementation/IMPLEMENTATION_PROGRESS.md`
  - `_sdd/implementation/IMPLEMENTATION_PROGRESS_PHASE_<n>.md` (if phased)
- Review artifacts generated by this skill:
  - `_sdd/implementation/IMPLEMENTATION_REPORT_PHASE_<n>.md`
  - `_sdd/implementation/IMPLEMENTATION_REPORT.md`
- Updated in-session execution state via `update_plan`.

## Workflow

### 1) Load and Validate Plan

- Locate the active plan file.
- Parse components, phases, tasks, dependencies, acceptance criteria, and open questions.
- If no plan source exists, request plan creation via `implementation-plan` or `feature-draft`.

### 2) Initialize Tracking

- Keep plan task IDs as canonical IDs.
- Initialize `update_plan` with the active phase and first unblocked task.
- If needed, create/refresh a concise task table in progress markdown.

Recommended table columns:

| ID | Task | Priority | Dependencies | Status | Tests |
|----|------|----------|--------------|--------|-------|

### 3) Execute by Phase and Dependency

- Work phase-by-phase unless the user explicitly requests a different order.
- Pick unblocked tasks first.
- Prioritize `P0` before `P1/P2/P3` within the same dependency level.

### 4) Run TDD Per Acceptance Criterion

For each criterion in the active task:

1. RED: Write a failing test that proves missing behavior.
2. GREEN: Implement the smallest change that passes the test.
3. REFACTOR: Improve code structure without changing behavior.
4. Re-run relevant tests after each step.

Do not batch multiple criteria into one opaque change.

### 5) Close Task with Evidence

Before marking task complete:

- Every acceptance criterion has test coverage or explicit rationale.
- Newly added tests pass.
- Related regression tests pass.
- Progress file includes what changed, where, and test results.

Then set task status to `completed` in both markdown progress and `update_plan`.

### 6) Phase Review (Built-in)

After each phase, run cross-cutting quality checks using `references/review-checklist.md`.

Review categories:

- Security
- Error handling
- Code patterns
- Performance
- Test quality
- Cross-task integration

Classify findings:

- Critical: must fix before moving to next phase
- Quality: document and proceed
- Improvement: optional follow-up

Critical findings must be fixed with TDD:

1. write a test that reproduces the issue
2. implement fix
3. re-run tests and phase checks

Save phase review to `_sdd/implementation/IMPLEMENTATION_REPORT_PHASE_<n>.md`.

### 7) Final Review and Combined Report

After all phases, run cross-phase review and generate `_sdd/implementation/IMPLEMENTATION_REPORT.md`.

Combined report sections:

1. Progress summary
2. Phase review summary
3. Cross-phase findings
4. Issue table with severity and status
5. Recommendations
6. Final conclusion (`READY`, `NEEDS_WORK`, `BLOCKED`)

## Progress Report Contract

Include these sections in each progress update:

1. Scope covered (phase/task IDs)
2. Files changed
3. Tests added/updated and pass status
4. Blockers and decisions needed
5. Next task(s)

## Phase Review Contract

For each phase review, include:

1. files touched in phase
2. review checklist summary by category
3. issue severity table
4. gate decision (proceed/fix-first)

## When to Pause and Ask

Ask the user directly when:

- A requirement is ambiguous and affects behavior.
- A dependency blocks all available tasks.
- A proposed change appears out-of-scope versus plan.
- Test environment cannot be reproduced from `_sdd/env.md`.
- Multiple plan sources conflict in scope.

## Integration

- Upstream planning: `implementation-plan`, `feature-draft` (Part 2)
- Built-in reviews: phase review + final review inside this skill
- Standalone audit option: `implementation-review`
- Spec sync if drift found: `spec-update-done`
