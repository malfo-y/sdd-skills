# Routing and Matching

## Input Routing

### 1. Wiki only

1. 위키 페이지를 먼저 가져온다.
2. 제목, 본문 상단 메타, 날짜/시간 표현을 추출한다.
3. 해당 시간대의 Calendar 이벤트를 찾는다.
4. 이벤트가 있으면 관련 Google Docs 회의록을 연결한다.

### 2. Meet only

1. Meet 링크에서 Meet 코드를 추출한다.
2. Calendar 이벤트를 먼저 찾는다.
3. 이벤트 제목/시간으로 Drive 문서를 찾는다.
4. 관련 위키 회의록이 있으면 연결한다.

### 3. Calendar only

1. 이벤트를 기준점으로 고정한다.
2. 설명, 제목, 시간대, 첨부 링크를 읽는다.
3. Drive 문서와 위키 페이지를 연결한다.

### 4. Docs only

1. 문서를 먼저 읽는다.
2. 제목, 생성/수정 시각, 본문 첫 부분의 회의 식별 단서를 추출한다.
3. Calendar 이벤트와 위키 회의록을 역추적한다.

### 5. Drive file only

1. Drive URL에서 파일 ID와 파일 유형을 먼저 식별한다.
2. 메타데이터와 가능한 본문 텍스트를 확보한다.
3. 제목, 생성/수정 시각, 본문 첫 부분의 회의 식별 단서를 추출한다.
4. Calendar 이벤트와 위키 회의록을 역추적한다.

### 6. Multiple inputs

1. 먼저 같은 회의 이벤트인지 판단한다.
2. 동일 회의로 보이면 이벤트 축을 고정한다.
3. 위키, 캘린더, 문서를 같은 회의 아래 병합한다.
4. 서로 다른 회의로 보이면 후보 목록과 근거를 먼저 제시한다.

## Matching Priority

기본 우선순위:

1. 시간 일치
2. Meet 코드 일치
3. 제목 유사도
4. 사용자 확인

`Docs/Drive` 생성/수정 시각은 1-3단계를 보강하는 부가 힌트로만 사용한다. 독립 fallback 단계로 승격하지 않는다.

## Confidence Heuristic

- High: 시간과 Meet 코드가 모두 일치
- Medium: 시간 또는 제목 유사도가 강하게 일치
- Low: 단일 문서 또는 Drive 파일 단서만 있고 이벤트/위키 연결이 약함

## Candidate Presentation

후보가 여러 개면 상위 2-3개만 보여준다.

각 후보에는 아래를 포함한다.

- 제목
- 날짜/시간
- 어떤 단서로 연결했는지
- 왜 불확실한지
