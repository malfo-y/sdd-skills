# Guide Create Output Format

이 문서는 `guide-create` 스킬이 생성하는 기능 가이드 문서의 형식 규칙을 정의한다.

## 1. 파일 규칙

- 출력 디렉토리: `_sdd/guides/`
- 파일명: `guide_<slug>.md`
- `slug` 규칙:
  - 영문 소문자, 숫자, `_`만 사용
  - 공백, 하이픈, 슬래시는 `_`로 정규화
  - 길이는 가능하면 50자 이하
  - 기능명이 한국어만 있는 경우에도 영문 또는 음차 기반 slug를 우선 시도하고, 어렵다면 의미 기반 영문 요약 slug를 사용

## 2. 백업 규칙

- 대상 파일이 이미 존재하면 새 파일을 쓰기 전에 `_sdd/guides/prev/`에 백업한다.
- 백업 파일명: `PREV_guide_<slug>_<timestamp>.md`
- timestamp 형식: `YYYYMMDD_HHMMSS`

## 3. 문서 필수 구조

```markdown
# 기능 가이드: <feature name>

**생성일**: YYYY-MM-DD
**입력 소스**: conversation / spec / code / mixed
**대상 기능**: <feature name>
**신뢰도**: High / Medium / Low

## 설명
[기능 목적, 사용자 가치, 범위, 선행 조건]

## 규칙
[구현 규칙, 인터페이스 규칙, 상태/에러/데이터 처리 규칙]

## 체크리스트
[구현 전 / 구현 중 / 리뷰 전후]

## 예시
[좋은 예시 중심, 필요 시 안티패턴]
```

## 4. 섹션 작성 규칙

### 설명

반드시 포함할 내용:

- 이 기능이 해결하는 문제
- 사용자 또는 운영 관점의 가치
- 이번 가이드가 다루는 범위
- 구현/테스트/의존성 전제

### 규칙

최소한 다음 범주를 고려한다:

- 구현 규칙
- 인터페이스 또는 API 규칙
- 상태 변화 규칙
- 에러 처리 규칙
- 데이터 검증/저장 규칙

확정 근거가 약한 경우에는 "권장", "추정", "가정" 표현을 사용한다.

### 체크리스트

다음 세 묶음을 권장한다:

- 구현 전 체크
- 구현 중 체크
- 완료/리뷰 전 체크

각 항목은 검증 가능한 문장으로 적는다.

### 예시

최소 1개의 긍정 예시를 포함한다.

가능하면 다음 중 하나 이상을 포함한다:

- 요청/응답 예시
- 상태 전이 예시
- UI/UX 동작 예시
- 테스트 관점 예시
- 안티패턴 예시

## 5. 부록 규칙

부록은 선택이지만 가능한 한 포함한다.

권장 부록:

### 관련 스펙 레퍼런스

- 파일 경로만 적지 말고 가능한 경우 섹션명까지 같이 적는다.
- 예:
  - `_sdd/spec/my_project.md` → `목표 > 결제 승인`
  - `_sdd/spec/my_project_API.md` → `API Reference > POST /payments/confirm`

### 관련 코드 레퍼런스

- 파일 경로와 심볼명을 함께 적는다.
- 예:
  - `src/payments/service.ts` - `confirmPayment`
  - `src/payments/schema.ts` - `ConfirmPaymentRequest`

### 가정 및 미확정 사항

- 근거가 부족한 부분을 숨기지 말고 명시한다.
- 예:
  - "결제 만료 정책은 스펙에서 직접 확인되지 않아 15분 만료로 가정함"

## 6. 신뢰도 규칙

- **High**: 핵심 목적/범위/규칙이 spec+code 모두에서 확인됨
- **Medium**: 목적과 범위는 spec에서 확인되나 세부 규칙 일부가 추론임
- **Low**: spec 근거가 약하거나 코드 근거가 거의 없어 가정 비중이 큼

## 7. 다중 기능 입력 처리

- 입력에서 여러 기능이 감지되면 기능별로 문서를 분리한다.
- 한 문서에 여러 기능을 억지로 합치지 않는다.
- 파일명 충돌 가능성이 있으면 slug 뒤에 더 구체적인 도메인 단어를 붙인다.
