# Discussion Question Guide

토론 스킬에서 사용하는 질문 전략 및 `request_user_input` 설계 가이드.

## 토론 단계별 질문 전략

### Phase 1: 탐색 (Exploration) - Round 1-3

목적: 토픽의 범위와 맥락을 이해한다.

| 질문 유형 | 템플릿 | 사용 시점 |
|-----------|--------|----------|
| 동기 확인 | "이 [토픽]을 고려하게 된 계기가 무엇인가요?" | Round 1 |
| 현황 파악 | "현재 [관련 영역]은 어떤 상태인가요?" | Round 1-2 |
| 목표 정의 | "이상적인 결과는 어떤 모습인가요?" | Round 2-3 |
| 제약 조건 | "꼭 지켜야 할 제약 조건이 있나요?" | Round 2-3 |

### Phase 2: 분석 (Analysis) - Round 4-6

목적: 대안을 비교하고 트레이드오프를 분석한다.

| 질문 유형 | 템플릿 | 사용 시점 |
|-----------|--------|----------|
| 대안 비교 | "A와 B 중 [기준]에서 어떤 것이 더 적합할까요?" | Round 4-5 |
| 트레이드오프 | "[장점]을 위해 [단점]을 감수할 수 있나요?" | Round 4-5 |
| 우선순위 | "가장 중요한 기준은 무엇인가요?" | Round 5-6 |
| 반론 제시 | "반대로 [대안]을 선택한다면 어떤 이점이 있을까요?" | Round 5-6 |

### Phase 3: 수렴 (Convergence) - Round 7+

목적: 결론을 도출하고 합의를 형성한다.

| 질문 유형 | 템플릿 | 사용 시점 |
|-----------|--------|----------|
| 결론 확인 | "지금까지 논의를 바탕으로 [결론]으로 정리할까요?" | Round 7+ |
| 위험 확인 | "이 결정의 가장 큰 위험은 무엇이라고 보시나요?" | Round 7+ |
| 실행 확인 | "다음 단계로 어떤 행동이 필요할까요?" | Round 8+ |
| 최종 합의 | "이 방향으로 정리해도 괜찮을까요?" | 마지막 라운드 |

## `request_user_input` 설계 원칙

### 질문 수

- 한 라운드에 질문은 정확히 1개만 사용한다.

### 옵션 수

- 옵션은 총 2-3개만 사용한다.
- 권장 패턴은 `실질 선택지 2개 + 정리/종료 1개`다.
- `Other`는 명시적으로 만들지 않는다. 자유 입력은 UI가 자동 제공한다.

### 옵션 설계 기준

1. 상호 배타적이어야 한다.
2. 주요 선택지를 빠짐없이 커버해야 한다.
3. 의미가 분명해야 한다.
4. 마지막 선택지는 항상 `정리/종료`여야 한다.

### 추천 패턴

```text
question: "[현재 라운드 질문]"
options:
1. "[추천 선택지]" (Recommended)
2. "[대안 선택지]"
3. "정리/종료"
```

### 자유 입력 활용

자유 입력은 다음 상황에서 활용한다.

- 사용자가 예상 밖의 방향을 제시할 때
- 선택지보다 더 구체적인 제약을 직접 설명할 때
- "추가 리서치 필요" 같은 상태 신호를 줄 때

## 언어 처리 규칙

- 질문 본문, 옵션 설명, 맥락 요약, 중간 요약, 최종 요약은 모두 사용자의 활성 입력 언어를 따른다.
- 한국어, English, 그 외 사용자 입력 언어 모두 허용 대상이다.
- 사용자가 토론 도중 언어를 바꾸면 다음 질문부터 즉시 새 언어를 사용한다.
- 저장 파일 내용도 대화와 동일한 언어를 사용한다.
- 저장 언어를 별도로 지정한 경우에만 그 지시를 우선한다.

## 리서치 트리거 조건

### 자동 리서치 (Step 2)

- 코드베이스 관련 토픽 → 로컬 코드 탐색 자동 수행
- 아키텍처 토픽 → 로컬 구조 + 웹 리서치 병행
- 기술 비교 토픽 → 공식 문서와 1차 자료 중심 비교

### 토론 중 리서치 (Step 3)

- 사용자가 자유 입력으로 추가 리서치를 요청함
- 사용자의 주장에 대한 사실 확인이 필요함
- 특정 코드/파일에 대한 확인이 필요함
- 최신성 있는 비교 자료가 필요함

### 리서치 스킵 조건

- 동일 주제에 대해 이미 조사한 경우
- 토론이 수렴 단계에 진입한 경우
- 사용자가 빠른 정리를 원할 때

## 모드 제약

- `discussion` 스킬은 interactive skill이다.
- `request_user_input`을 사용할 수 없는 모드에서는 우회하지 않는다.
- 현재 모드에서 `request_user_input`이 불가능하면 토론을 시작하지 않고 즉시 중단한다.
