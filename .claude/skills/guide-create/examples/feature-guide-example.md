# 기능 가이드: 결제 승인

**생성일**: 2026-03-12
**입력 소스**: mixed (conversation + spec + code)
**대상 기능**: 결제 승인
**신뢰도**: Medium

## 설명

결제 승인 기능은 사용자가 결제 의사를 최종 확정했을 때 거래를 승인 상태로 전환하고, 후속 주문 처리 흐름을 시작하기 위한 핵심 기능이다. 이 기능의 사용자 가치는 결제 결과를 예측 가능하게 만들고, 주문/영수증/재고 반영 같은 후속 단계가 중복 없이 한 번만 실행되도록 보장하는 데 있다.

이 가이드는 "승인 요청 수신 → 유효성 검증 → 승인 처리 → 후속 이벤트 기록"까지를 다룬다. 결제 생성, 환불, 정산 배치는 범위에 포함하지 않는다. 스펙상 결제 상태 모델이 정의되어 있고 일부 서비스 코드가 존재한다고 가정한다.

## 규칙

1. 승인 요청은 이미 만료되었거나 취소된 결제에 대해 성공으로 처리하면 안 된다.
2. 동일 결제에 대한 중복 승인 요청은 멱등적으로 다루어야 하며, 이미 승인된 경우 기존 승인 결과를 재사용하는 편이 안전하다.
3. 외부 결제 게이트웨이 응답을 내부 상태 모델에 바로 노출하지 말고 도메인 상태로 변환한다.
4. 승인 성공 후 주문 생성이나 이벤트 발행이 이어진다면, 최소 한 번 이상 처리에 따른 중복 부작용을 막는 식별자를 함께 남긴다.
5. 에러 메시지는 사용자용 메시지와 내부 진단용 로그를 분리한다.
6. 금액, 통화, 결제 수단 식별자 등 핵심 입력 필드는 승인 전에 다시 검증한다.

## 체크리스트

### 구현 전

- [ ] 스펙에서 결제 상태 전이 규칙을 확인했다.
- [ ] 승인 API 또는 서비스 진입점이 어디인지 확인했다.
- [ ] 중복 승인 시 기대 동작을 코드 또는 스펙에서 확인했다.

### 구현 중

- [ ] 승인 가능 상태가 아닌 경우 실패 분기가 존재한다.
- [ ] 외부 응답을 내부 도메인 모델로 정규화한다.
- [ ] 멱등성 키 또는 중복 방지 장치가 존재한다.
- [ ] 성공/실패 로그에 추적 가능한 식별자가 포함된다.

### 완료/리뷰 전

- [ ] 정상 승인 케이스 테스트가 있다.
- [ ] 중복 승인 요청 테스트가 있다.
- [ ] 만료/취소 결제 실패 케이스 테스트가 있다.
- [ ] 후속 이벤트 또는 주문 생성이 중복 실행되지 않는지 점검했다.

## 예시

### 좋은 예시

사용자가 결제 확인 버튼을 누르면 서버는 현재 결제 상태가 `pending`인지 먼저 확인한다. 이후 외부 게이트웨이 승인 결과를 `approved` 또는 도메인 에러로 변환하고, 승인 이력이 이미 존재하는 경우 동일 결과를 반환한다. 승인 직후에는 주문 생성 이벤트를 기록하되, 동일 결제 ID에 대해 중복 이벤트가 생기지 않도록 처리한다.

### 안티패턴

- 외부 게이트웨이의 상태 문자열을 그대로 DB에 저장하는 방식
- 이미 `approved` 상태인 결제에 대해 매번 새 주문 생성 로직을 다시 실행하는 방식
- 사용자 응답에 내부 예외 메시지나 스택트레이스를 그대로 노출하는 방식

### 가정 및 미확정 사항

- 결제 승인 타임아웃은 스펙에서 직접 확인되지 않아 별도 운영 정책이 있다고 가정했다.
- 주문 생성은 비동기 이벤트 기반일 가능성이 높지만, 정확한 큐 이름이나 전달 스키마는 코드 근거가 없어서 확정하지 않았다.
