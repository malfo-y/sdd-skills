# Guide Create Output Format

이 문서는 `guide-create` 스킬이 생성하는 기능 기술 보고서의 형식 규칙을 정의한다.

## 1. 파일 규칙

- 출력 디렉토리: `_sdd/guides/`
- 파일명: `guide_<slug>.md`
- `slug` 규칙:
  - 영문 소문자, 숫자, `_`만 사용
  - 공백, 하이픈, 슬래시는 `_`로 정규화
  - 길이는 가능하면 50자 이하
  - 기능명이 한국어만 있는 경우에도 영문 또는 음차 기반 slug를 우선 시도하고, 어렵다면 의미 기반 영문 요약 slug를 사용

## 2. 백업 규칙

- 대상 파일이 이미 존재하면 새 파일을 쓰기 전에 `_sdd/guides/prev/`에 백업한다.
- 백업 파일명: `PREV_guide_<slug>_<timestamp>.md`
- timestamp 형식: `YYYYMMDD_HHMMSS`

## 3. 문서 필수 구조

```markdown
# 기능 기술 보고서: <feature name>

**생성일**: YYYY-MM-DD
**입력 소스**: conversation / spec / code / mixed
**대상 기능**: <feature name>
**신뢰도**: High / Medium / Low

## 1. 배경 및 동기
[이 기능이 해결하는 문제, 왜 필요한가, 대안 대비 선택 이유]

## 2. 핵심 설계
[핵심 아이디어/알고리즘, 설계 결정과 이유, 코드 citation]

## 3. 사용 시나리오 가이드
[시나리오별 end-to-end 사용 흐름, 정상/예외 시나리오]

## 4. API 레퍼런스
[인터페이스 상세, 파라미터, 리턴값, 에러, 코드 예시]

## 5. 구현 가이드
[핵심 규칙, 체크리스트, 안티패턴]
```

## 4. 섹션 작성 규칙

### 1. 배경 및 동기

반드시 포함할 내용:

- 이 기능이 해결하는 문제 (사용자/시스템 관점)
- 왜 이 기능이 필요한가 (동기와 맥락)
- 대안 대비 현재 접근의 선택 이유 (스펙에서 확인 가능한 경우)

스펙의 배경/동기 섹션이 있으면 이를 기능 관점으로 축약한다. 없으면 스펙의 문맥에서 추론하되 가정을 명시한다.

### 2. 핵심 설계

반드시 포함할 내용:

- 기능의 핵심 아이디어 또는 알고리즘 (어떻게 동작하는가)
- 주요 설계 결정과 그 이유
- 코드 citation: `[filepath:symbolName]` 형식

코드 citation이 있으면 해당 코드의 핵심 로직을 간결하게 설명한다. 코드 발췌는 10줄 이하로 핵심만 포함한다. 코드 근거가 없으면 스펙 기반으로 설명하고 "코드 미확인" 표기.

### 3. 사용 시나리오 가이드 ★특화

**이 섹션이 가이드의 핵심이다.** 가능한 한 구체적이고 자세하게 작성한다.

작성 원칙:

- 시나리오별로 서브섹션을 나눈다
- 각 시나리오는 다음 구조를 따른다:
  1. **전제 조건**: 이 시나리오가 성립하는 초기 상태
  2. **입력**: 구체적 데이터 예시 포함
  3. **처리 흐름**: 단계별 동작 설명
  4. **기대 결과**: 성공/실패 시 예상 출력
- 정상 시나리오를 먼저 작성하고, 예외/에러 시나리오를 이어서 작성한다
- 가능하면 요청/응답 JSON, 상태 전이, 코드 호출 예시 등을 구체적으로 포함한다

최소 요구사항:

- 정상 시나리오 1개 이상
- 예외/에러 시나리오 1개 이상

권장 시나리오 유형:

- 기본 정상 흐름 (happy path)
- 엣지 케이스 (경계값, 빈 입력 등)
- 에러/실패 시나리오 (권한 부족, 잘못된 입력, 외부 서비스 실패 등)
- 동시성/멱등성 시나리오 (해당되는 경우)

### 4. API 레퍼런스 ★특화

**이 섹션은 개별 인터페이스별 reference로 작성한다.** 가능한 한 구체적이고 자세하게 작성한다.

각 엔드포인트/함수/인터페이스에 대해:

- **시그니처**: 엔드포인트 URL, 함수 시그니처, 또는 인터페이스 정의
- **파라미터**: 테이블 형식 권장 (이름 | 타입 | 필수/선택 | 설명)
- **리턴값/응답**: 성공 응답 구조와 필드 설명
- **에러 코드**: 에러 상황별 코드/메시지 (테이블 형식 권장)
- **코드 예시**: 요청/응답 JSON 또는 함수 호출 예시

API가 없는 기능(내부 로직, UI 컴포넌트 등)인 경우:

- 함수/메서드 인터페이스를 동일한 구조로 문서화
- 또는 "이 기능은 외부 API가 없습니다" 명시 후 내부 인터페이스 문서화

### 5. 구현 가이드

다음 세 파트를 포함한다:

**핵심 규칙**: 구현 시 반드시 지켜야 할 규칙과 제약. 최소한 다음 범주를 고려:
- 상태 변화 규칙
- 에러 처리 규칙
- 데이터 검증/저장 규칙
- 확정 근거가 약한 경우 "권장", "추정", "가정" 표현을 사용

**체크리스트**: 세 묶음으로 나눈다:
- 구현 전 체크
- 구현 중 체크
- 완료/리뷰 전 체크
- 각 항목은 검증 가능한 문장

**안티패턴**: 피해야 할 구현 방식을 근거와 함께 기술

## 5. 부록 규칙

부록은 선택이지만 가능한 한 포함한다.

권장 부록:

### 관련 스펙 레퍼런스

- 파일 경로만 적지 말고 가능한 경우 섹션명까지 같이 적는다.
- 예:
  - `_sdd/spec/my_project.md` → `목표 > 결제 승인`
  - `_sdd/spec/my_project_API.md` → `API Reference > POST /payments/confirm`

### 관련 코드 레퍼런스

- 파일 경로와 심볼명을 함께 적는다.
- 예:
  - `src/payments/service.ts` - `confirmPayment`
  - `src/payments/schema.ts` - `ConfirmPaymentRequest`

### 가정 및 미확정 사항

- 근거가 부족한 부분을 숨기지 말고 명시한다.
- 예:
  - "결제 만료 정책은 스펙에서 직접 확인되지 않아 15분 만료로 가정함"

## 6. 신뢰도 규칙

- **High**: 배경/설계/시나리오/API가 spec+code 모두에서 확인됨
- **Medium**: 배경과 설계는 spec에서 확인되나 시나리오/API 세부가 일부 추론임
- **Low**: spec 근거가 약하거나 코드 근거가 거의 없어 가정 비중이 큼

## 7. 다중 기능 입력 처리

- 입력에서 여러 기능이 감지되면 기능별로 문서를 분리한다.
- 한 문서에 여러 기능을 억지로 합치지 않는다.
- 파일명 충돌 가능성이 있으면 slug 뒤에 더 구체적인 도메인 단어를 붙인다.
