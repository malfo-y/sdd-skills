# Scale Assessment -- 규모 판단 상세 가이드라인

autopilot이 Step 4에서 규모를 판단할 때 참조하는 상세 가이드라인이다.

---

## 판단 프로세스

```
1. 정량적 기준으로 초기 규모 산정
2. 정성적 기준으로 보정
3. 경계 사례 규칙 적용
4. 최종 규모 결정
5. 테스트 전략 결정
```

---

## 정량적 기준

### 영향 파일 수

코드베이스 탐색(Step 3) 결과에서 수정/생성이 필요한 파일 수를 기준으로 판단한다.

| 파일 수 | 규모 |
|---------|------|
| 1-3개 | 소규모 |
| 4-10개 | 중규모 |
| 10개+ | 대규모 |

**카운트 방법:**
- 직접 수정하는 코드 파일만 카운트 (설정 파일, 문서 제외)
- 테스트 파일은 별도 카운트하지 않음 (테스트는 구현에 포함)
- 자동 생성 파일(migration, build artifact)은 제외

### 신규 컴포넌트 수

새로 생성해야 하는 독립적 컴포넌트(모듈, 클래스, 서비스)의 수.

| 컴포넌트 수 | 규모 |
|------------|------|
| 0-1개 | 소규모 |
| 1-3개 | 중규모 |
| 3개+ | 대규모 |

### 스펙 변경 규모

| 스펙 변경 | 규모 |
|----------|------|
| 없음 | 소규모 |
| 기존 섹션 수정/패치 | 중규모 |
| 신규 섹션 추가 | 대규모 |

### 예상 코드 줄 수

대략적인 추정치로 보조 기준으로만 사용한다.

| 코드 줄 | 규모 |
|---------|------|
| < 200줄 | 소규모 |
| 200-1000줄 | 중규모 |
| 1000줄+ | 대규모 |

## 정성적 기준

### 복잡도 레벨

| 레벨 | 설명 | 규모 |
|------|------|------|
| Low | 단일 함수/메서드 수정, 로직 변경 | 소규모 |
| Medium | 여러 모듈 간 연동, 새 API 엔드포인트 | 중규모 |
| High | 아키텍처 레벨 변경, 새 시스템 도입 | 대규모 |

### 의존성

| 의존성 | 규모 |
|--------|------|
| 기존 코드 내부 수정만 | 소규모 |
| 새 라이브러리/패키지 1-2개 | 중규모 |
| 새 의존성 3개+ 또는 외부 서비스 연동 | 대규모 |

### 리스크

| 리스크 | 규모 |
|--------|------|
| 로컬 변경, 다른 모듈에 영향 없음 | 소규모 |
| 모듈 간 영향 있음, 회귀 테스트 필요 | 중규모 |
| 시스템 전체 영향, 데이터 마이그레이션 필요 | 대규모 |

## 경계 사례 처리

### 규칙 1: 정량 vs 정성 불일치

정량적 기준과 정성적 기준이 다른 규모를 가리키면 **더 큰 규모를 선택**한다.

예시:
- 파일 3개(소규모)지만 아키텍처 변경(대규모) → **대규모**
- 파일 5개(중규모)지만 단순 패턴 반복(소규모) → **중규모** (정량 기준 우선)

### 규칙 2: 확신 없음

어느 규모에도 명확히 해당하지 않으면 **중규모를 기본값**으로 사용한다.
중규모 파이프라인은 소규모에도, 대규모의 축소판으로도 유연하게 동작한다.

### 규칙 3: 사용자 의도 반영

- "빠르게", "간단히", "최소한으로" → 규모를 한 단계 **낮추는** 것을 고려
- "완벽하게", "프로덕션 레벨로", "전체적으로" → 규모를 한 단계 **높이는** 것을 고려

### 규칙 4: 순수 신규 생성

기존 코드 수정 없이 순수 신규 파일만 생성하는 경우:
- 파일 수 기준을 한 단계 완화 (신규 생성은 수정보다 리스크가 낮음)
- 5개 신규 파일 → 중규모 (아닌 대규모)

### 규칙 5: 버그 수정

버그 수정 요청은 영향 범위를 면밀히 확인한다:
- 단일 파일 버그 수정 → 소규모
- 여러 파일에 걸친 버그 수정 → 중규모
- 시스템적 버그 (설계 결함) → 대규모

## 예시별 규모 판단

### 소규모 예시

| 요청 | 판단 근거 |
|------|----------|
| "로그인 실패 시 에러 메시지 수정" | 파일 1개, 문자열 변경, 스펙 변경 없음 |
| "API 응답에 timestamp 필드 추가" | 파일 1-2개, 모델+핸들러 수정, 스펙 변경 없음 |
| "테스트 커버리지 추가" | 테스트 파일만 추가, 코드 변경 없음 |
| "설정값 변경 (타임아웃 30초→60초)" | 설정 파일 1개 |

### 중규모 예시

| 요청 | 판단 근거 |
|------|----------|
| "사용자 프로필 페이지 추가" | 모델+뷰+라우팅+테스트 (4-6파일), 기존 스펙에 섹션 패치 |
| "파일 업로드 기능 구현" | 핸들러+스토리지+검증+UI (5-7파일), 새 의존성 1개 |
| "이메일 알림 시스템 추가" | 서비스+템플릿+설정 (4-5파일), 외부 서비스 연동 1개 |
| "기존 REST API에 GraphQL 엔드포인트 추가" | 스키마+리졸버+테스트 (5-8파일) |

### 대규모 예시

| 요청 | 판단 근거 |
|------|----------|
| "JWT 기반 인증 시스템 구현" | 모델+미들웨어+라우트+테스트+문서 (15+파일), 신규 스펙 섹션 |
| "다국어 지원 추가" | 전체 UI 수정+번역 시스템 (20+파일), 아키텍처 변경 |
| "마이크로서비스 분리" | 서비스 분할+통신+배포 (30+파일), 아키텍처 변경 |
| "실시간 알림 시스템 (WebSocket)" | 새 프로토콜+서버+클라이언트 (10+파일), 새 인프라 |

## 테스트 전략 판단

### 인라인 디버깅 선택 기준

다음 조건을 **모두** 만족하면 인라인 디버깅:
1. 테스트 1회 실행이 수 분 이내
2. 테스트 환경이 로컬에서 완결
3. 외부 서비스 의존성 없음 (또는 mock 가능)

### ralph-loop-init 선택 기준

다음 조건 중 **하나라도** 해당하면 ralph-loop-init:
1. 테스트 1회 실행이 수십 분 이상
2. GPU 또는 대용량 리소스 필요 (ML 학습 등)
3. 외부 CI/CD 파이프라인을 통해 검증 필요
4. 빌드 시간이 10분 이상

### 판단 힌트

| 프로젝트 유형 | 기본 전략 |
|-------------|----------|
| 웹 백엔드 (Node.js, Python) | 인라인 디버깅 |
| 프론트엔드 (React, Vue) | 인라인 디버깅 |
| CLI 도구 | 인라인 디버깅 |
| ML/DL 프로젝트 | ralph-loop-init |
| 모바일 앱 (빌드 필요) | ralph-loop-init |
| 인프라/DevOps | ralph-loop-init |
