# Sample Orchestrator

이 파일은 autopilot이 생성하는 오케스트레이터의 품질 기준 예시다.

review가 포함된 small/medium/large 모든 path에서는 `sdd-skills:implementation-agent`와 `sdd-skills:implementation-review-agent`가 항상 subagent mapping으로 실행된다. 경로가 단순하더라도 부모 autopilot이 local implementation/review로 대체하지 않는다.

## Example A: single-phase medium direct path

사용자 요청: `/autopilot JWT 기반 인증 시스템 구현`

```markdown
# Orchestrator: JWT 인증 시스템

**생성일**: 2026-04-10T14:30:00
**규모**: 중규모
**생성자**: autopilot

## 기능 설명

JWT 기반 인증 시스템을 구현한다. 로그인, 로그아웃, 토큰 갱신 기능을 포함한다.

## Acceptance Criteria
- [ ] 로그인 성공 시 access/refresh token이 발급된다.
- [ ] refresh token 갱신이 동작하고 만료 정책이 적용된다.
- [ ] 로그아웃 시 refresh token이 무효화된다.
- [ ] 관련 검증이 테스트 또는 리뷰 evidence와 연결된다.

## Reasoning Trace

- trivial/small direct-only 변경만 `sdd-skills:feature-draft-agent`를 생략할 수 있다. 이 요청은 non-trivial change이므로 planning entry는 `sdd-skills:feature-draft-agent`로 시작한다.
- feature draft Part 2가 task/dependency/validation을 충분히 제공하고 single-phase이므로 `sdd-skills:implementation-plan-agent`는 생략한다.
- feature draft output은 downstream implementation 입력으로 소비되기 전에 `sdd-skills:plan-review-agent` producer review gate를 통과해야 한다.
- single-phase medium path라 `Review-Fix Loop.scope`는 `global`로 유지하되, `implementation` 직후 즉시 completion gate로 닫는다.
- review가 포함된 direct path이므로 구현과 리뷰 모두 subagent mapping으로 유지한다.
- 테스트는 짧은 API/서비스 검증 중심이라 인라인 전략을 선택한다.

## Pipeline Steps

### Step 1: feature-draft
**Claude subagent_type**: `sdd-skills:feature-draft-agent`
**입력 파일**: `_sdd/spec/main.md`
**출력 파일**: `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`

**프롬프트**:
JWT 기반 인증 시스템에 대한 feature draft를 작성하세요.
Part 1에는 temporary spec 7섹션을 포함하고, `Contract/Invariant Delta`와 `Validation Plan`을 ID로 연결하세요.
Part 2에는 implementation이 직접 읽을 수 있는 Target Files, dependency, validation detail을 포함하세요.

### Step 2: feature-draft producer review gate
**Claude subagent_type**: `sdd-skills:plan-review-agent`
**입력 파일**:
- `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`
- `_sdd/spec/main.md`
**출력 파일**: `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md` (accepted or regenerated by producer)

**프롬프트**:
방금 생성된 feature draft output을 downstream implementation 입력으로 소비하기 전에 검토하세요.
temporary spec 7섹션, `Contract/Invariant Delta`, `Validation Plan` linkage, Target Files/dependency/validation detail이 implementation 입력으로 충분한지 확인하세요.
gate 실패 시 finding을 fix task로 normalize하지 말고 `sdd-skills:feature-draft-agent` 산출물을 reject/regenerate 대상으로 보고하세요.

### Step 3: implementation dispatch controller
**Claude subagent_type**: `sdd-skills:implementation-agent`
**입력 파일**:
- `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`
- `_sdd/spec/main.md`
**출력 파일**:
- `src/middleware/auth.js`
- `src/routes/auth.js`
- `src/services/auth-service.js`
- `src/services/token-service.js`
- `src/models/user.js`
- `test/services/auth-service.test.js`

**프롬프트**:
feature draft를 기반으로 구현을 진행하세요.
temporary spec의 `Contract/Invariant Delta`와 `Validation Plan`을 기준으로 TDD 방식으로 진행하세요.

**Dispatch controller semantics**: 이 `sdd-skills:implementation-agent` step은 feature 전체를 한 leaf에 넘기는 generic single call이 아니라 autopilot이 실행하는 dispatch controller다. autopilot은 이 step의 task를 dependency 기반 **병렬 dispatch 그룹**("dependency 없음 + Target Files disjoint → 병렬")으로 파생해 task당 `sdd-skills:implementation-agent` leaf를 dispatch한다(병렬 불가·저확신 시 순차). 이 병렬 dispatch 그룹은 아래 review-fix gate의 Checkpoint 리뷰 그룹과 다른 중첩 개념이다 (각각 "무엇을 동시에 dispatch하나" vs "언제 gate를 닫나"). progress/report는 autopilot이 canonical 경로로 소유한다.

**Immediate review-fix gate (must finish before Step 4)**:
- `scope`: `global`
- `max_rounds`: 3
- `exit_condition`: `critical = 0 AND high = 0 AND medium = 0`
- `fix_targets`: `critical/high/medium`
- `low_policy`: `low` findings are advisory/logged follow-up and do not block this gate.
- `timing`: Step 3 `sdd-skills:implementation-agent` dispatch controller 직후 즉시 실행하는 completion gate
- `agent_mapping`: `review = sdd-skills:implementation-review-agent`, `fix = sdd-skills:implementation-agent`, `re-review = sdd-skills:implementation-review-agent`
- `execution_sequence`: `sdd-skills:implementation-agent -> sdd-skills:implementation-review-agent -> sdd-skills:implementation-agent (if needed) -> sdd-skills:implementation-review-agent`
- autopilot은 Step 3 구현 직후 같은 범위로 `sdd-skills:implementation-review-agent` subagent를 즉시 호출한다.
- review 입력에는 최소한 `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`, `_sdd/spec/main.md`, 현재 변경 파일 목록, 관련 테스트 결과를 포함한다.
- `review invocation prompt contract`:
  - "방금 끝난 JWT 인증 구현 범위만 검토하세요. 응답은 findings first여야 하며, 각 finding은 severity, 파일/라인, 관련 Acceptance Criteria 또는 temporary spec linkage, 근거, 권장 수정 방향을 포함해야 합니다."
- `fix invocation prompt contract`:
  - "직전 `implementation-review` finding 중 `critical/high/medium`만 닫으세요. unrelated dirty changes를 되돌리지 말고, 수정 후 관련 테스트가 다시 실행 가능한 상태를 유지하세요."
- `re-review invocation prompt contract`:
  - "직전 review finding이 실제로 해소되었는지 먼저 검증하고, 같은 범위에서 새 `critical/high/medium`이 남는지만 다시 보고하세요. findings first 형식을 유지하세요."
- `sdd-skills:implementation-review-agent` 결과에 `critical/high/medium`이 하나라도 있으면 autopilot은 fix를 수행한다 — finding을 fix-task로 보고 **finding 하나씩 순차로** `sdd-skills:implementation-agent` leaf를 dispatch한다(finding 영향 파일 = leaf Target Files). 초기 구현의 병렬 dispatch 그룹과 달리 fix는 순차다(별도 fix 분해 기계장치 없음).
- fix 후 autopilot은 같은 scope로 `sdd-skills:implementation-review-agent` subagent를 재호출한다.
- 이 gate와 required inline validation이 모두 닫힌 뒤에만 Step 4로 진행한다.

### Step 4: spec-update-done
**Claude subagent_type**: `sdd-skills:spec-update-done-agent`
**입력 파일**:
- `_sdd/spec/main.md`
- `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`
- 구현된 코드 파일
**출력 파일**: `_sdd/spec/main.md`

**프롬프트**:
JWT 인증 시스템 구현 완료 기준으로 global spec을 실제 코드와 동기화하세요.
temporary spec의 실행 정보는 버리고, 구현되어 검증된 persistent repo-wide information만 global spec에 반영하세요.

**Precondition**:
Step 3에 연결된 immediate review-fix gate와 required inline validation이 모두 닫힌 뒤에만 실행한다.

## Review-Fix Loop

- 이 오케스트레이터의 authoritative 순서와 프롬프트 계약은 Step 3의 `Immediate review-fix gate`에 인라인으로 고정한다.
- 별도 후처리 loop는 없고, Step 3 gate가 닫히기 전에는 `spec-update-done`으로 진행할 수 없다.

## Test Strategy

- `mode`: `inline`
- `commands`: 관련 서비스 테스트 + 인증 라우트 smoke 검증
- `rationale`: 변경 범위가 서비스/라우트 단위에 닫혀 있어 인라인 검증만으로도 `Validation Plan`을 닫을 수 있다.
- `reporting`: 통과/실패 건수와 잔여 수동 확인 항목을 `_sdd/pipeline/report_<topic>_<timestamp>.md`에 기록

## Error Handling

- `retries`: feature-draft/plan-review/spec-update-done는 1회, implementation/review-fix loop는 최대 3회 재시도
- `critical_steps`: Step 3 immediate review-fix gate, inline test
- `non_critical_steps`: spec wording polish
- `failure_policy`: review-fix gate를 닫지 못하거나 inline test가 실패하면 Step 4로 진행하지 않고 보고서와 로그에 blocker를 남긴다
```

## Example B: multi-phase expanded path

사용자 요청: `/autopilot 결제 시스템 전체 구현`

```markdown
# Orchestrator: 결제 시스템

**생성일**: 2026-04-10T16:00:00
**규모**: 대규모
**생성자**: autopilot

## 기능 설명

Stripe 기반 결제 시스템을 구현한다. 결제 생성, 웹훅 처리, 환불 로직, 운영 상태 동기화를 포함한다.

## Acceptance Criteria
- [ ] 결제 생성과 상태 전환이 Stripe 계약과 일치한다.
- [ ] 웹훅 서명 검증과 재시도 처리가 구현된다.
- [ ] 환불/실패/보상 흐름이 테스트 또는 리뷰 evidence로 검증된다.
- [ ] multi-phase implementation에서 group exit criteria와 최종 통합 리뷰가 모두 충족된다.

## Reasoning Trace

- trivial/small direct-only 변경만 `sdd-skills:feature-draft-agent`를 생략할 수 있다. 이 요청은 non-trivial planning이 필요하므로 entry는 `sdd-skills:feature-draft-agent`로 고정한다.
- planned persistent global alignment가 있으므로 `spec-update-todo`를 조건부로 추가한다.
- 구현 범위가 크고 dependency chain이 깊은 multi-phase 실행이므로 `sdd-skills:implementation-plan-agent`로 phase/task breakdown을 반드시 확장한다.
- feature draft와 implementation plan output은 각각 downstream 소비 전에 `sdd-skills:plan-review-agent` producer review gate를 통과해야 한다.
- Step 4 generation 시점에는 orchestrator file만 생성하고 phase count/boundary는 아직 runtime metadata라서, downstream `implementation` step을 flat single-shot으로 쓰지 않고 `Execution Mode: phase-iterative` + `Phase Source`로 선언한다.
- multi-phase plan이므로 `Review-Fix Loop.scope = per-group`을 사용한다. `implementation-plan`의 phase `Checkpoint` 필드가 group boundary를 결정한다. Phase 1(domain model/Stripe abstraction)은 후속 2개 phase가 의존하므로 `Checkpoint=true`(단독 group), Phase 2+3은 `Checkpoint=false`+`Checkpoint=true`로 한 group에 묶인다. 최종적으로 2개 그룹이므로 마지막 group gate 후 별도 final integration review 1회를 추가한다.

## Pipeline Steps

### Step 1: feature-draft
**Claude subagent_type**: `sdd-skills:feature-draft-agent`
**입력 파일**:
- `_sdd/spec/main.md`
- `_sdd/spec/components.md`
**출력 파일**: `_sdd/drafts/2026-04-10_feature_draft_payment_system.md`

**프롬프트**:
Stripe 기반 결제 시스템에 대한 feature draft를 작성하세요.
Part 1에는 temporary spec 7섹션과 `Contract/Invariant Delta`, `Validation Plan` ID linkage를 포함하세요.
Part 2에는 후속 `implementation-plan`이 phase/task 수준으로 확장할 수 있도록 Target Files, dependency, validation detail을 남기세요.

### Step 2: feature-draft producer review gate
**Claude subagent_type**: `sdd-skills:plan-review-agent`
**입력 파일**:
- `_sdd/drafts/2026-04-10_feature_draft_payment_system.md`
- `_sdd/spec/main.md`
**출력 파일**: `_sdd/drafts/2026-04-10_feature_draft_payment_system.md` (accepted or regenerated by producer)

**프롬프트**:
방금 생성된 feature draft output을 `sdd-skills:spec-update-todo-agent`와 `sdd-skills:implementation-plan-agent` 입력으로 소비하기 전에 검토하세요.
temporary spec 7섹션, `Contract/Invariant Delta`, `Validation Plan` linkage, downstream planning 입력 적합성을 확인하세요.
gate 실패 시 finding을 fix task로 normalize하지 말고 `sdd-skills:feature-draft-agent` 산출물을 reject/regenerate 대상으로 보고하세요.

### Step 3: spec-update-todo
**Claude subagent_type**: `sdd-skills:spec-update-todo-agent`
**입력 파일**:
- `_sdd/spec/main.md`
- `_sdd/drafts/2026-04-10_feature_draft_payment_system.md`
**출력 파일**: `_sdd/spec/main.md`

**프롬프트**:
결제 시스템 구현 전에 필요한 persistent global alignment만 global spec에 반영하세요.
temporary execution detail은 넣지 말고, 이후 `implementation-plan`과 `implementation`이 읽어야 할 repo-wide invariant만 정리하세요.

### Step 4: implementation-plan
**Claude subagent_type**: `sdd-skills:implementation-plan-agent`
**입력 파일**:
- `_sdd/drafts/2026-04-10_feature_draft_payment_system.md`
- `_sdd/spec/main.md`
**출력 파일**: `_sdd/implementation/2026-04-10_implementation_plan_payment_system.md`

**프롬프트**:
feature draft 이후 확장 단계로서 multi-phase implementation plan을 작성하세요.
각 phase마다 `goal`, `task set / dependency closure`, `validation focus`, `exit criteria`, `carry-over policy`, `checkpoint`를 명시하고, downstream `implementation`이 phase-iterative로 소비할 수 있게 구조화하세요.

### Step 5: implementation-plan producer review gate
**Claude subagent_type**: `sdd-skills:plan-review-agent`
**입력 파일**:
- `_sdd/implementation/2026-04-10_implementation_plan_payment_system.md`
- `_sdd/drafts/2026-04-10_feature_draft_payment_system.md`
- `_sdd/spec/main.md`
**출력 파일**: `_sdd/implementation/2026-04-10_implementation_plan_payment_system.md` (accepted or regenerated by producer)

**프롬프트**:
방금 생성된 implementation plan output을 downstream implementation controller 입력으로 소비하기 전에 검토하세요.
`Contract/Invariant Delta` coverage, task/Target Files, dependency/checkpoint/carry-over 필드, validation linkage, phase-iterative controller 입력 적합성을 확인하세요.
gate 실패 시 finding을 fix task로 normalize하지 말고 `sdd-skills:implementation-plan-agent` 산출물을 reject/regenerate 대상으로 보고하세요.

### Step 6: implementation dispatch controller
**Claude subagent_type**: `sdd-skills:implementation-agent`
**Execution Mode**: `phase-iterative`
**Phase Source**: `_sdd/implementation/2026-04-10_implementation_plan_payment_system.md`
**입력 파일**:
- `_sdd/implementation/2026-04-10_implementation_plan_payment_system.md`
- `_sdd/spec/main.md`
**출력 파일**:
- 결제 서비스/웹훅/환불 관련 코드 및 테스트 파일

**프롬프트**:
`Phase Source`의 현재 phase만 읽고 해당 phase의 `goal`, `task set / dependency closure`, `validation focus`, `exit criteria`, `carry-over policy`, `checkpoint`를 기준으로 구현하세요.
`Checkpoint=false` phase는 light validation(test/typecheck/exit criteria) 후 다음 phase로 진행합니다. `Checkpoint=true` phase 직후에는 같은 group 범위의 review-fix gate를 즉시 닫아야 합니다. gate가 열려 있으면 다음 group으로 진행하지 마세요.

**Dispatch controller semantics (phase 내부)**: 이 `sdd-skills:implementation-agent` step은 phase 전체를 한 leaf에 넘기는 generic single call이 아니라 autopilot이 실행하는 dispatch controller다. autopilot은 각 phase 안에서 task를 dependency 기반 **병렬 dispatch 그룹**("dependency 없음 + Target Files disjoint → 병렬")으로 파생해 task당 `sdd-skills:implementation-agent` leaf를 dispatch한다(병렬 불가·저확신 시 순차). 이 **병렬 dispatch 그룹(phase 내부)**은 아래 `group boundary`의 **Checkpoint 리뷰 그룹(phase 경계)**과 다른 중첩 개념이다 — 전자는 "무엇을 동시에 dispatch하나", 후자는 "언제 review-fix gate를 닫나". progress/report는 autopilot이 canonical 경로로 소유한다.

**Per-group review-fix gate (must finish before Step 7)**:
- `scope`: `per-group`
- `max_rounds_per_group`: 3
- `exit_condition`: `critical = 0 AND high = 0 AND medium = 0`
- `fix_targets`: `critical/high/medium`
- `low_policy`: `low` findings are advisory/logged follow-up and do not block this gate.
- `timing`: 각 `Checkpoint=true` phase 직후 즉시 실행하는 completion gate. 마지막 phase는 implicit `Checkpoint=true`.
- `Phase Source`: `_sdd/implementation/2026-04-10_implementation_plan_payment_system.md`
- `group boundary`:
  - Group 1: Phase 1 (Checkpoint=true — 후속 2개 phase가 domain model에 의존)
  - Group 2: Phase 2 (Checkpoint=false) + Phase 3 (Checkpoint=true — 마지막 phase)
- `mid-group emergency`: Group 내 phase light validation에서 `critical` 이슈 발견 시 그 즉시 해당 group의 review-fix gate를 forced early 트리거
- `group exit criteria`:
  - Group 1 (Phase 1): 결제 도메인 모델과 Stripe client abstraction이 테스트 통과 상태로 고정된다.
  - Group 2 (Phase 2+3): 결제 생성/승인/실패 흐름, 웹훅 검증, 환불/운영 보정 흐름과 observability가 통합 검증된다.
- `carry-over policy`:
  - Default: `None`
  - `critical/high/medium` 이슈는 group exit를 막는다.
  - 예외 carry-over는 plan에 명시된 항목만 허용하고, 로그에 근거와 후속 group absorb point를 남긴다.
- `agent_mapping`:
  - `review = sdd-skills:implementation-review-agent`
  - `fix = sdd-skills:implementation-agent`
  - `re-review = sdd-skills:implementation-review-agent`
- `execution_sequence_per_group`: `[Checkpoint=false phases: light validation only] -> Checkpoint=true phase sdd-skills:implementation-agent -> sdd-skills:implementation-review-agent -> sdd-skills:implementation-agent (if needed) -> sdd-skills:implementation-review-agent -> group validation`
- autopilot은 각 Checkpoint phase 직후 해당 group 전체 범위로 `sdd-skills:implementation-review-agent` subagent를 즉시 호출한다.
- review 입력에는 최소한 `_sdd/implementation/2026-04-10_implementation_plan_payment_system.md`, `_sdd/drafts/2026-04-10_feature_draft_payment_system.md`, 해당 group의 전체 변경 파일, focused test 결과를 포함한다.
- `review invocation prompt contract`:
```
