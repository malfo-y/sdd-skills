# Sample Orchestrator

이 파일은 autopilot이 생성하는 오케스트레이터의 품질 기준 예시다.

review가 포함된 small/medium/large 모든 path에서는 `sdd-skills:implementation-agent`와 correctness(`sdd-skills:implementation-review-agent`) ∥ simplicity(`sdd-skills:simplicity-review-agent`) 2-reviewer가 항상 subagent mapping으로 실행된다(review는 두 reviewer 병렬, exit는 두 report의 합집합). 경로가 단순하더라도 부모 autopilot이 local implementation/review로 대체하지 않는다.

## Example A: single-phase medium path

사용자 요청: `/autopilot JWT 기반 인증 시스템 구현`

```markdown
# Orchestrator: JWT 인증 시스템

**생성일**: 2026-04-10T14:30:00
**규모**: 중규모
**생성자**: autopilot

## 기능 설명

JWT 기반 인증 시스템을 구현한다. 로그인, 로그아웃, 토큰 갱신 기능을 포함한다.

## Acceptance Criteria
- [ ] 로그인 성공 시 access/refresh token이 발급된다.
- [ ] refresh token 갱신이 동작하고 만료 정책이 적용된다.
- [ ] 로그아웃 시 refresh token이 무효화된다.
- [ ] 관련 검증이 테스트 또는 리뷰 evidence와 연결된다.

## Reasoning Trace

- trivial/small direct-only 변경만 `sdd-skills:feature-draft-agent`를 생략할 수 있다. 이 요청은 non-trivial change이므로 planning entry는 `sdd-skills:feature-draft-agent`로 시작한다.
- feature-draft Part 2는 dependency·phase가 없는 flat task-set을 산출하므로, 구현 직전 `sdd-skills:task-ordering-agent`가 dependency·phase를 파생한다. 이 요청은 변경 범위가 좁아 phase 1개(single-phase)로 수렴한다.
- `sdd-skills:task-ordering-agent`는 autopilot이 소유하는 ordering 파생 step이며 producer가 아니라 review-fix gate가 면제된다(self-check만, plan-review gate 없음). 산출물은 downstream `implementation` step에 `Phase Source`로 hand-off된다.
- feature draft output은 downstream 소비 전에 `sdd-skills:plan-review-agent` producer review gate를 통과해야 한다.
- single-phase(phase 1개)이므로 `Review-Fix Loop.scope`는 `global`로 유지하고, `implementation` 직후 즉시 completion gate로 닫는다(per-group 불필요).
- review가 포함된 path이므로 구현과 리뷰 모두 subagent mapping으로 유지한다.
- 테스트는 짧은 API/서비스 검증 중심이라 인라인 전략을 선택한다.

## Pipeline Steps

### Step 1: feature-draft
**Claude subagent_type**: `sdd-skills:feature-draft-agent`
**입력 파일**: `_sdd/spec/main.md`
**출력 파일**: `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`

**프롬프트**:
JWT 기반 인증 시스템에 대한 feature draft를 작성하세요.
Part 1은 `Part 1: Spec Delta`로 작성하고, `Change Summary`, `Scope Delta`, `Persistent Spec Implications`만 포함하세요.
Part 2는 `Part 2: Implementation and Validation Plan`으로 작성하고, `Contract/Invariant Delta and Coverage` thin index와 task별 `Contracts`/`Validation`·Target Files를 포함한 **flat task-set**으로 작성하세요 (dependency·phase는 후속 `sdd-skills:task-ordering-agent`가 파생하므로 포함하지 마세요).
top-level `Risks/Mitigations and Open Questions`에는 `Risks and Mitigations`를 `Risk / Impact / Mitigation` 표로, `Open Questions`를 Decision / Alternatives / Confidence / User confirmation needed 스키마로 작성하세요.

### Step 2: feature-draft producer review gate
**Claude subagent_type**: `sdd-skills:plan-review-agent`
**입력 파일**:
- `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`
- `_sdd/spec/main.md`
**출력 파일**: `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md` (accepted or regenerated by producer)

**프롬프트**:
방금 생성된 feature draft output을 downstream implementation 입력으로 소비하기 전에 검토하세요.
`Part 1: Spec Delta`의 marker/3-section slim shape를 확인하세요.
`Part 2: Implementation and Validation Plan`의 coverage index(고아 delta 없음)와 task별 `Contracts`/`Validation`·Target Files가 flat task-set으로서 task-ordering 입력에 충분한지 확인하세요.
top-level `Risks/Mitigations and Open Questions`가 risk table과 decision/assumption surface로 충분한지 확인하세요.
gate 실패 시 finding을 fix task로 normalize하지 말고 `sdd-skills:feature-draft-agent` 산출물을 reject/regenerate 대상으로 보고하세요.

### Step 3: task-ordering
**Claude subagent_type**: `sdd-skills:task-ordering-agent`
**입력 파일**:
- `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`
- `_sdd/spec/main.md`
**출력 파일**: `없음 (transient final response)`

**프롬프트**:
지정된 feature-draft flat task-set만 읽고 dependency·topological parallel wave·single/multi-phase·review checkpoint를 판단하세요.
파일을 만들거나 task 정의를 전사하지 말고 `Status`, `Source`, `Mode`, `Execution`, `Dependencies`, `Checkpoints`, `Notes`만 담은 최종 Markdown을 반환하세요.

### Step 4: implementation
**Step kind**: `implementation-dispatch-controller`
**Stage agents**: `sdd-skills:test-author-agent` (Stage A 테스트 작성) · orchestrator RED gate (Stage B) · `sdd-skills:implementation-agent` (Stage C GREEN→REFACTOR)
**Execution Mode**: `phase-iterative`
**Phase Source**: `task_ordering.response`
**입력 파일**:
- `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`
- `_sdd/spec/main.md`
**출력 파일**:
- `src/middleware/auth.js`
- `src/routes/auth.js`
- `src/services/auth-service.js`
- `src/services/token-service.js`
- `src/models/user.js`
- `test/services/auth-service.test.js`

**프롬프트**:
`task_ordering.response`의 `Execution` phase를 wave로 사용하고, `Source` feature draft에서 해당 task의 AC·Target Files·Validation Plan을 읽으세요. (a)/(b) task는 autopilot이 닫은 RED 게이트가 넘긴 고정 실패 테스트 + RED 증거를 입력으로, (c) test-free로 triage된 task는 고정 테스트·RED 증거 대신 triage 근거를 입력으로 받아 GREEN→REFACTOR만 수행하세요(RED는 자체 수행하지 않습니다).

**Step kind 실행 방식 (`implementation-dispatch-controller`)**: autopilot은 `task_ordering.response`의 각 `Execution` phase를 병렬 wave로 소비하고 `Source`에서 task 본문을 결합한다. fan-out 직전 file-disjoint를 검증한 뒤 각 wave를 3-way triage(test/structural-check/test-free 분류) → test-author 병렬 dispatch → orchestrator RED gate → implementation 병렬 dispatch 순서로 실행한다. test-free로 분류된 task는 test-author/RED를 스킵하고 triage 근거를 입력으로 곧장 implementation dispatch로 넘긴다. `Checkpoints`는 wave와 별개의 review boundary이며 progress/report는 autopilot이 소유한다.

**Immediate review-fix gate (must finish before Step 5)**:
- `scope`: `global`
- `max_rounds`: 3
- `exit_condition`: `critical = 0 AND high = 0 AND medium = 0`
- `fix_targets`: `critical/high/medium`
- `low_policy`: `low` findings are advisory/logged follow-up and do not block this gate.
- `timing`: Step 4 implementation step 직후 즉시 실행하는 completion gate
- `agent_mapping`:
  - `review = sdd-skills:implementation-review-agent`
  - `review = sdd-skills:simplicity-review-agent`
  - `re-review = sdd-skills:implementation-review-agent`
  - `re-review = sdd-skills:simplicity-review-agent`
  - `fix = sdd-skills:implementation-agent`
  - `fix (correctness, test-first) = sdd-skills:test-author-agent -> orchestrator RED 게이트 -> sdd-skills:implementation-agent`
  - `fix (simplicity/refactor, direct) = sdd-skills:implementation-agent`
- `execution_sequence`: `sdd-skills:implementation-agent -> [sdd-skills:implementation-review-agent ∥ sdd-skills:simplicity-review-agent] -> sdd-skills:implementation-agent (if needed) -> [sdd-skills:implementation-review-agent ∥ sdd-skills:simplicity-review-agent]` (두 reviewer 병렬, exit는 두 report의 합집합)
- autopilot은 Step 4 구현 직후 같은 범위로 correctness(`sdd-skills:implementation-review-agent`) ∥ simplicity(`sdd-skills:simplicity-review-agent`) 2-reviewer subagent를 즉시 병렬 호출한다(exit는 두 report의 합집합).
- review 입력에는 최소한 `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`, `_sdd/spec/main.md`, 현재 변경 파일 목록, 관련 테스트 결과를 포함한다.
- `review invocation prompt contract`:
  - "방금 끝난 JWT 인증 구현 범위만 검토하세요. 응답은 findings first여야 하며, 각 finding은 severity, 파일/라인, 관련 Acceptance Criteria 또는 temporary spec linkage, 근거, 권장 수정 방향을 포함해야 합니다."
- `fix invocation prompt contract`:
  - correctness finding(동작 버그): "이 버그를 노출하는 실패 테스트가 먼저 작성되고(`sdd-skills:test-author-agent` dispatch + orchestrator RED 게이트로 실패 확인) 고정 실패 테스트 + RED 증거가 입력으로 전달됩니다. 직전 두 reviewer finding 합집합 중 `critical/high/medium`만 닫고, 테스트를 수정하지 말고 GREEN→REFACTOR로 fix하세요. unrelated dirty changes를 되돌리지 말고, 수정 후 관련 테스트가 다시 실행 가능한 상태를 유지하세요."
  - simplicity/refactor finding: "실패 테스트 없이 직접 fix하세요. 직전 두 reviewer finding 합집합 중 `critical/high/medium`만 닫고, unrelated dirty changes를 되돌리지 말고, 수정 후 관련 테스트가 다시 실행 가능한 상태를 유지하세요."
- `re-review invocation prompt contract`:
  - "correctness ∥ simplicity 각 reviewer에 해당 lens의 기존 review 리포트 경로를 전달해 re-review mode로 진입하세요. 새 리포트를 만들지 말고, 직전 review finding이 실제로 해소됐는지 검증한 뒤 같은 범위에서 새 `critical/high/medium`이 남는지 판정해, 각 lens 기존 리포트의 `Current Status`를 갱신하고 `Iteration History`에 이번 회차(resolved/still-open/new)를 append하세요. findings first 형식 유지."
- 두 reviewer(`sdd-skills:implementation-review-agent` correctness ∥ `sdd-skills:simplicity-review-agent` simplicity) report의 **합집합**에 `critical/high/medium`이 하나라도 있으면 autopilot은 fix를 수행한다 — finding을 fix-task로 보고 **finding 하나씩 순차로** dispatch한다(finding 영향 파일 = leaf Target Files). correctness finding은 test-first다: 먼저 `sdd-skills:test-author-agent` leaf로 그 버그를 노출하는 실패 테스트를 작성하고 orchestrator RED 게이트로 실패를 확인한 뒤 고정 실패 테스트 + RED 증거를 입력으로 `sdd-skills:implementation-agent` leaf를 재dispatch한다. simplicity/refactor finding은 실패 테스트 없이 `sdd-skills:implementation-agent` leaf로 직접 fix한다. 초기 구현의 병렬 dispatch 그룹과 달리 fix는 순차다(별도 fix 분해 기계장치 없음).
- fix 후 autopilot은 같은 scope로 두 reviewer(correctness ∥ simplicity)를 다시 병렬 재호출한다(exit는 두 report의 합집합).
- 이 gate와 required inline validation이 모두 닫힌 뒤에만 Step 5로 진행한다.

### Step 5: spec-sync (post-implementation)
**Claude subagent_type**: `sdd-skills:spec-sync-agent`
**입력 파일**:
- `_sdd/spec/main.md`
- `_sdd/drafts/2026-04-10_feature_draft_jwt_auth.md`
- 구현된 코드 파일
**출력 파일**: `_sdd/spec/main.md`

**프롬프트**:
JWT 인증 시스템 구현 완료 기준으로 global spec을 실제 코드와 동기화하세요.
temporary spec의 실행 정보는 버리고, 구현되어 검증된 persistent repo-wide information만 global spec에 반영하세요.

**Precondition**:
Step 4에 연결된 immediate review-fix gate와 required inline validation이 모두 닫힌 뒤에만 실행한다.

## Review-Fix Loop

- 이 오케스트레이터의 authoritative 순서와 프롬프트 계약은 Step 4의 `Immediate review-fix gate`에 인라인으로 고정한다.
- 별도 후처리 loop는 없고, Step 4 gate가 닫히기 전에는 `spec-sync`(post-implementation 호출)로 진행할 수 없다.

## Test Strategy

- `mode`: `inline`
- `commands`: 관련 서비스 테스트 + 인증 라우트 smoke 검증
- `rationale`: 변경 범위가 서비스/라우트 단위에 닫혀 있어 인라인 검증만으로도 `Validation Plan`을 닫을 수 있다.
- `reporting`: 통과/실패 건수와 잔여 수동 확인 항목을 `_sdd/pipeline/report_<topic>_<timestamp>.md`에 기록

## Error Handling

- `retries`: feature-draft/plan-review/task-ordering/spec-sync는 1회, implementation/review-fix loop는 최대 3회 재시도
- `critical_steps`: Step 4 immediate review-fix gate, inline test
- `non_critical_steps`: spec wording polish
- `failure_policy`: review-fix gate를 닫지 못하거나 inline test가 실패하면 Step 5로 진행하지 않고 보고서와 로그에 blocker를 남긴다
```

## Example B: multi-phase expanded path

사용자 요청: `/autopilot 결제 시스템 전체 구현`

```markdown
# Orchestrator: 결제 시스템

**생성일**: 2026-04-10T16:00:00
**규모**: 대규모
**생성자**: autopilot

## 기능 설명

Stripe 기반 결제 시스템을 구현한다. 결제 생성, 웹훅 처리, 환불 로직, 운영 상태 동기화를 포함한다.

## Acceptance Criteria
- [ ] 결제 생성과 상태 전환이 Stripe 계약과 일치한다.
- [ ] 웹훅 서명 검증과 재시도 처리가 구현된다.
- [ ] 환불/실패/보상 흐름이 테스트 또는 리뷰 evidence로 검증된다.
- [ ] multi-phase implementation에서 group exit criteria와 최종 통합 리뷰가 모두 충족된다.

## Reasoning Trace

- trivial/small direct-only 변경만 `sdd-skills:feature-draft-agent`를 생략할 수 있다. 이 요청은 non-trivial planning이 필요하므로 entry는 `sdd-skills:feature-draft-agent`로 고정한다.
- planned persistent global alignment가 있으므로 `spec-sync`(planned 호출)를 조건부로 추가한다. 동일 `spec-sync`는 구현 전(planned 반영)과 구현 후(sync)로 최대 2회 호출될 수 있다.
- 구현 범위가 크고 dependency chain이 깊으므로 구현 직전 `sdd-skills:task-ordering-agent`가 feature-draft flat task-set에서 dependency와 topological parallel wave를 파생하고 별도 `Checkpoints`에 review boundary를 반환한다(multi-phase 판정). autopilot은 final Markdown을 `task_ordering.response`로 보존한다.
- feature draft output은 downstream 소비 전에 `sdd-skills:plan-review-agent` producer review gate를 통과해야 한다. `sdd-skills:task-ordering-agent`는 producer가 아니라 ordering 파생 step이므로 review-fix gate가 면제되고 self-check만 수행한다(plan-review gate 없음).
- Step 4 generation 시점에는 orchestrator file만 생성하고 phase count/boundary는 아직 runtime metadata라서, downstream `implementation` step을 flat single-shot으로 쓰지 않고 `Execution Mode: phase-iterative` + `Phase Source`로 선언한다.
- `task_ordering.response`가 multi-phase이고 `Checkpoints`에 Phase 1이 중간 경계로 지정되므로 `Review-Fix Loop.scope = per-group`을 사용한다. Phase 1은 단독 group, Phase 2+3은 마지막 implicit checkpoint까지 한 group이다. 그룹이 2개이므로 마지막 group gate 후 별도 final integration review 1회를 추가한다.

**Step kind**: `implementation-dispatch-controller`
**Stage agents**: `sdd-skills:test-author-agent` (Stage A 테스트 작성) · orchestrator RED gate (Stage B) · `sdd-skills:implementation-agent` (Stage C GREEN→REFACTOR)

**Phase Source**: `task_ordering.response`
**입력 파일**:

**출력 파일**:

**프롬프트**:
`task_ordering.response`의 `Execution` phase를 wave로 사용하고, `Source` feature draft에서 해당 task의 AC·Target Files·Validation Plan을 읽으세요. response의 `Checkpoints`에 지정된 phase와 마지막 implicit checkpoint에서 review-fix gate를 닫으세요.

**Step kind 실행 방식 (`implementation-dispatch-controller`)**: autopilot은 `task_ordering.response`의 각 `Execution` phase를 topological parallel wave로 소비하고 `Source` feature draft에서 task 본문을 결합한다. fan-out 직전 file-disjoint를 검증한 뒤 각 wave를 3-way triage(test/structural-check/test-free 분류) → test-author 병렬 dispatch → orchestrator RED gate → implementation 병렬 dispatch 순서로 실행한다. test-free로 분류된 task는 test-author/RED를 스킵하고 triage 근거를 입력으로 곧장 implementation dispatch로 넘긴다. wave끼리는 순차이며 cross-wave 중첩은 없다. 별도 `Checkpoints` 목록은 review boundary로만 해석하고 progress/report는 autopilot이 소유한다.

**Per-group review-fix gate (must finish before Step 6)**:
- `scope`: `per-group`
- `max_rounds_per_group`: 3
- `exit_condition`: `critical = 0 AND high = 0 AND medium = 0`
- `fix_targets`: `critical/high/medium`
- `low_policy`: `low` findings are advisory/logged follow-up and do not block this gate.
- `timing`: `task_ordering.response`의 별도 `Checkpoints`에 지정된 phase 직후 즉시 실행한다. 마지막 phase는 목록 기재 여부와 무관하게 implicit checkpoint다.
- `Phase Source`: `task_ordering.response`
- `group boundary`:
  - Group 1: Phase 1 (`Checkpoints`에 명시 — 후속 2개 phase가 domain model에 의존)
  - Group 2: Phase 2 + Phase 3 (Phase 3은 마지막 implicit checkpoint)
- `mid-group emergency`: Group 내 phase light validation에서 `critical` 이슈 발견 시 그 즉시 해당 group의 review-fix gate를 forced early 트리거
- `group exit criteria`:
  - Group 1 (Phase 1): 결제 도메인 모델과 Stripe client abstraction이 테스트 통과 상태로 고정된다.
  - Group 2 (Phase 2+3): 결제 생성/승인/실패 흐름, 웹훅 검증, 환불/운영 보정 흐름과 observability가 통합 검증된다.
- `carry-over policy`:
  - Default: `None`
  - `critical/high/medium` 이슈는 group exit를 막는다.
  - 예외 carry-over는 plan에 명시된 항목만 허용하고, 로그에 근거와 후속 group absorb point를 남긴다.
- `agent_mapping`:
  - `review = sdd-skills:implementation-review-agent`
  - `review = sdd-skills:simplicity-review-agent`
  - `re-review = sdd-skills:implementation-review-agent`
  - `re-review = sdd-skills:simplicity-review-agent`
  - `fix = sdd-skills:implementation-agent`
  - `fix (correctness, test-first) = sdd-skills:test-author-agent -> orchestrator RED 게이트 -> sdd-skills:implementation-agent`
  - `fix (simplicity/refactor, direct) = sdd-skills:implementation-agent`

- `review invocation prompt contract`:
  - "현재 group 범위(Phase N~M) 전체를 검토하세요. 응답은 findings first여야 하며, 각 finding은 severity, 파일/라인, 관련 group task 또는 temporary spec linkage, 근거, 권장 수정 방향을 포함해야 합니다."
- `fix invocation prompt contract`:
  - correctness finding(동작 버그): "이 버그를 노출하는 실패 테스트가 먼저 작성되고(`sdd-skills:test-author-agent` dispatch + orchestrator RED 게이트로 실패 확인) 고정 실패 테스트 + RED 증거가 입력으로 전달됩니다. 직전 두 reviewer finding 합집합 중 현재 group 범위의 `critical/high/medium`만 닫고, 테스트를 수정하지 말고 GREEN→REFACTOR로 fix하세요. unrelated dirty changes를 되돌리지 말고, 수정 후 관련 테스트가 다시 실행 가능한 상태를 유지하세요."
  - simplicity/refactor finding: "실패 테스트 없이 직접 fix하세요. 직전 두 reviewer finding 합집합 중 현재 group 범위의 `critical/high/medium`만 닫고, unrelated dirty changes를 되돌리지 말고, 수정 후 관련 테스트가 다시 실행 가능한 상태를 유지하세요."
- `re-review invocation prompt contract`:
  - "correctness ∥ simplicity 각 reviewer에 해당 lens의 기존 review 리포트 경로를 전달해 re-review mode로 진입하세요. 새 리포트를 만들지 말고, 직전 review finding이 실제로 해소됐는지 검증한 뒤 같은 group 범위에서 새 `critical/high/medium`이 남는지 판정해, 각 lens 기존 리포트의 `Current Status`를 갱신하고 `Iteration History`에 이번 회차(resolved/still-open/new)를 append하세요. findings first 형식 유지."
- 두 reviewer(`sdd-skills:implementation-review-agent` correctness ∥ `sdd-skills:simplicity-review-agent` simplicity) report의 **합집합**에 `critical/high/medium`이 하나라도 있으면 autopilot은 fix를 수행한다 — finding을 fix-task로 보고 **finding 하나씩 순차로** dispatch한다(finding 영향 파일 = leaf Target Files). correctness finding은 test-first다: 먼저 `sdd-skills:test-author-agent` leaf로 그 버그를 노출하는 실패 테스트를 작성하고 orchestrator RED 게이트로 실패를 확인한 뒤 고정 실패 테스트 + RED 증거를 입력으로 `sdd-skills:implementation-agent` leaf를 재dispatch한다. simplicity/refactor finding은 실패 테스트 없이 `sdd-skills:implementation-agent` leaf로 직접 fix한다. 초기 구현의 병렬 dispatch 그룹과 달리 fix는 순차다(별도 fix 분해 기계장치 없음).
- 현재 group gate가 닫히기 전에는 다음 group 또는 Step 6로 진행하지 않는다.

**Final integration review (must finish before Step 6) — 그룹 2개이므로 별도 실행**:

- `final integration review prompt contract`:
  - "group 경계를 넘는 회귀와 남은 cross-group gap을 검토하세요. 응답은 findings first 형식이며, overall integration readiness와 spec sync readiness를 함께 평가해야 합니다."
- 모든 group gate, 마지막 `final integration review`, required inline test가 모두 닫힌 뒤에만 Step 6로 진행한다.
```
