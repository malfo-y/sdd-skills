# Question Guide for Spec Draft

유형별 상세 질문 가이드입니다. 각 유형에 맞는 질문을 순서대로 진행하여 완전한 정보를 수집합니다.

---

## 새 기능 (New Feature)

### 필수 질문

1. **기능 이름**
   - "이 기능의 이름은 무엇인가요?"
   - 짧고 명확한 이름 권장 (예: "실시간 알림", "배치 처리")

2. **우선순위**
   - "이 기능의 우선순위는 어떻게 되나요?"
   - 선택지: High (다음 릴리스 필수) / Medium (계획된 개선) / Low (있으면 좋음)

3. **설명**
   - "이 기능이 무엇을 해야 하나요?"
   - 사용자 관점에서 기능의 동작 설명

4. **수용 기준 (Acceptance Criteria)**
   - "이 기능이 완료되었다고 판단할 수 있는 조건은 무엇인가요?"
   - 체크리스트 형태로 구체적인 조건 수집
   - 최소 2개 이상 권장

### 선택 질문

5. **카테고리**
   - "어떤 카테고리에 해당하나요?"
   - 선택지: Core Feature / Enhancement / Performance / Security / UI/UX

6. **대상 컴포넌트**
   - "어떤 파일이나 컴포넌트가 관련되나요?"
   - 기존 파일명 또는 새로 생성될 파일명

7. **기술적 참고사항**
   - "기술적으로 고려해야 할 사항이 있나요?"
   - 사용할 라이브러리, 아키텍처 제약 등

8. **의존성**
   - "이 기능을 구현하기 전에 완료되어야 하는 다른 작업이 있나요?"
   - 선행 작업이나 필요 조건

## 개선 사항 (Improvement)

### 필수 질문

1. **개선 대상**
   - "어떤 부분을 개선하고 싶나요?"
   - 현재 기능이나 동작 명시

2. **현재 상태**
   - "현재는 어떻게 동작하나요?"
   - 현재의 문제점이나 한계

3. **제안 내용**
   - "어떻게 개선되면 좋겠나요?"
   - 원하는 결과 상태

4. **개선 이유**
   - "왜 이 개선이 필요한가요?"
   - 비즈니스 가치나 기술적 이유

### 선택 질문

5. **우선순위**
   - "이 개선의 우선순위는 어떻게 되나요?"
   - 선택지: High / Medium / Low

6. **영향 범위**
   - "이 개선이 영향을 미치는 범위는 어디까지인가요?"
   - 관련 컴포넌트, 파일 목록

## 버그 리포트 (Bug Report)

### 필수 질문

1. **버그 이름/제목**
   - "버그를 한 문장으로 설명해주세요."
   - 명확하고 검색 가능한 제목

2. **심각도**
   - "이 버그의 심각도는 어떻게 되나요?"
   - 선택지:
     - High (시스템 사용 불가, 데이터 손실 위험)
     - Medium (기능 일부 제한, 우회 방법 존재)
     - Low (불편함, 미관상 문제)

3. **버그 설명**
   - "어떤 문제가 발생하나요?"
   - 증상과 영향 설명

4. **재현 방법**
   - "어떻게 하면 이 버그를 재현할 수 있나요?"
   - 단계별 재현 절차

5. **예상 동작**
   - "정상적으로 동작한다면 어떻게 되어야 하나요?"
   - 올바른 결과 명시

### 선택 질문

6. **발생 위치**
   - "이 버그가 발생하는 파일이나 위치를 알고 있나요?"
   - 파일명:라인번호 형식

7. **우회 방법**
   - "현재 이 문제를 피할 수 있는 방법이 있나요?"
   - 임시 해결책

8. **환경 정보**
   - "어떤 환경에서 발생하나요?"
   - OS, Python 버전, 의존성 버전 등

9. **오류 메시지**
   - "오류 메시지나 스택 트레이스가 있나요?"
   - 전체 오류 내용

## 컴포넌트 변경 (Component Change)

### 새 컴포넌트 추가

1. **컴포넌트 이름**
   - "새 컴포넌트의 이름은 무엇인가요?"
   - 파일명 또는 클래스명

2. **목적**
   - "이 컴포넌트의 목적은 무엇인가요?"
   - 단일 책임 원칙에 따른 명확한 역할

3. **입력**
   - "이 컴포넌트가 받는 입력은 무엇인가요?"
   - 데이터 타입, 형식

4. **출력**
   - "이 컴포넌트가 반환하는 출력은 무엇인가요?"
   - 데이터 타입, 형식

5. **주요 메서드**
   - "이 컴포넌트의 주요 메서드/함수는 무엇인가요?"
   - 함수명과 간단한 설명

### 기존 컴포넌트 수정

1. **대상 컴포넌트**
   - "어떤 컴포넌트를 수정하나요?"
   - 파일명 또는 클래스명

2. **변경 유형**
   - "어떤 종류의 변경인가요?"
   - 선택지: Enhancement (기능 추가) / Refactor (구조 개선) / Fix (수정)

3. **변경 내용**
   - "구체적으로 어떤 변경이 필요한가요?"
   - 추가/수정/삭제되는 부분

4. **호환성**
   - "기존 코드와의 호환성에 영향이 있나요?"
   - Breaking change 여부

## 설정 변경 (Configuration)

### 필수 질문

1. **설정 이름**
   - "설정 항목의 이름은 무엇인가요?"
   - 환경 변수명 또는 설정 키

2. **설정 유형**
   - "어떤 유형의 설정인가요?"
   - 선택지: Environment Variable / Config File / Command Line Argument

3. **설명**
   - "이 설정은 무엇을 제어하나요?"
   - 설정의 역할 설명

### 선택 질문

4. **필수 여부**
   - "이 설정은 필수인가요?"
   - 선택지: Yes (없으면 오류) / No (기본값 사용)

5. **기본값**
   - "기본값은 무엇인가요?"
   - 설정하지 않았을 때의 값

6. **유효 범위**
   - "유효한 값의 범위나 목록이 있나요?"
   - 숫자 범위, 허용되는 문자열 목록 등

7. **예시**
   - "사용 예시를 보여주세요."
   - 실제 설정 예시

## 질문 진행 팁

### 효과적인 질문 순서

1. 필수 질문 먼저 완료
2. 사용자 응답에 따라 관련 선택 질문 추가
3. 불명확한 부분은 후속 질문으로 명확화

### 응답 검증

- 너무 모호한 응답: "좀 더 구체적으로 설명해주실 수 있나요?"
- 기술적 정보 부족: "관련 파일이나 코드를 알고 계시나요?"
- 우선순위 미정: "다른 작업과 비교했을 때 얼마나 급한가요?"

### 대화 마무리

1. 수집된 정보 요약 표시
2. 누락된 정보 확인
3. 추가 항목 여부 질문
4. 다음 단계 안내 (`/spec-update-todo` 실행)
5. 중요한 결정이 있었으면 `_sdd/spec/DECISION_LOG.md`에 남길 항목 초안 정리
