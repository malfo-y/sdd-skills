# 토론 요약: spec-update-todo/done 멀티파일 스펙 배치 문제

**날짜**: 2026-03-20
**라운드 수**: 4
**참여 방식**: 구조화된 토론 (discussion skill)

## 핵심 논점 (Key Discussion Points)

1. **현재 갭**: spec-update-todo/done이 섹션-시맨틱 매핑만 있고 파일 레벨 배치 로직이 없어서, 새 기능이 기존 파일에 안 맞으면 아무 데나 붙여 넣어 어색한 스펙을 생성함.
2. **해결 방향**: 기존 파일에 의미적으로 안 맞는 변경사항은 새 서브 스펙 파일을 자동 생성하도록 허용.
3. **분리 기준**: 하이브리드 — 기본은 컴포넌트/기능 단위, 범용 내용(환경변수, 글로벌 설정 등)은 섹션(§) 기준.
4. **판단 로직**: 의미적 유사도 기반으로 LLM이 기존 파일 매칭 여부를 판단. 매칭 파일 없으면 새 파일 생성.

## 미결 질문 (Open Questions)

- [x] 빈 파일 방지: 새 파일 생성 후 내용이 너무 짧으면 (예: 3줄) 기존 파일에 병합하는 게 나은지 → **해결: 너무 작은 파일은 기존 파일에 병합한다. 애초에 너무 작은 파일이 생기는 것 자체가 분리 판단이 잘못된 것이므로, 최소 내용량 기준을 두고 미달 시 기존 파일에 포함시킨다.**

## 리서치 결과 요약 (Research Findings)

- **현재 구조**: 두 스킬 모두 섹션-시맨틱 매핑만 존재. 파일 탐색은 main.md → 링크된 서브파일 순서.
- **spec-rewrite 참고**: 소(단일)/중(index+component)/대(index+subdirs) 분할 기준 있음. update 스킬과 정합 필요.
- **핵심 갭**: "어떤 파일에 넣을지" 판단 로직 자체가 없음. 에이전트 재량에 100% 의존.
