# 토론 요약: SDD 스펙 문서의 정의 — 화이트페이퍼 스타일로 진화

**날짜**: 2026-03-13
**라운드 수**: 9
**참여 방식**: 구조화된 토론 (discussion skill)

## 핵심 논점 (Key Discussion Points)

1. **스펙 = 화이트페이퍼**: 스펙 문서는 학술 논문(화이트페이퍼/테크니컬 리포트)과 같은 구조를 가져야 한다. 배경, 동기, 핵심 아이디어, 알고리즘 설명이 포함되어야 한다.
2. **논문↔코드 매핑**: 논문의 각 요소가 코드 세계에 1:1로 대응된다 — citation은 코드 레퍼런스, 수식은 코드 스니펫, implementation detail은 구현 디테일, 결과는 기대 동작.
3. **현재 SDD의 한계**: 현재 SDD 스펙은 "소프트웨어 매뉴얼" 성격이 강하다 (컴포넌트 목록, API 나열, 설정값). 배경/동기, 핵심 아이디어 서사, 알고리즘 설명, 사용법 가이드가 모두 부족하다.
4. **진화 방향**: 기존 포맷을 대체하지 않고, 빠진 논문적 요소를 기존 섹션에 녹여 보강하는 "진화" 접근.
5. **하이브리드 Citation**: 핵심 알고리즘/설계 근거는 본문 인라인 citation, 기타 구현은 하단 Source 테이블. 중요도에 따라 분리.
6. **실제 코드 발취**: pseudocode가 아닌 실제 핵심 함수/클래스를 스펙에 직접 삽입. AI 인덱싱에 최적.
7. **사용 가이드 = 실험/결과 통합**: 논문의 Experiments & Results를 "시나리오별 사용법 + 예상 결과"로 통합. 스펙이 동시에 튜토리얼이자 검증 문서가 된다.
8. **Drift 관리**: spec-update-done을 강화하여 스펙 속 코드 스니펫과 실제 코드를 자동 대조/업데이트.

### 확정된 스펙 구조

```
# 프로젝트명 스펙

## 1. 배경 및 동기 (Introduction / Motivation)     ← NEW
   - 이 프로젝트가 해결하는 문제
   - 왜 이 접근법인가 (대안 비교)
   - 핵심 가치 제안

## 2. 핵심 설계 (Core Design)                       ← 아키텍처 개요 진화
   - 핵심 아이디어 서사 설명
   - 알고리즘/로직 흐름 + 실제 코드 발취 [inline citation]
   - 설계 근거 (왜 이 구조인가)

## 3. 아키텍처 상세                                  ← 기존 유지
## 4. 컴포넌트 상세                                  ← 기존 유지 + 코드 스니펫 보강

## 5. 사용 가이드 & 기대 결과 (Usage & Results)      ← 통합 NEW
   - 시나리오 1: [이렇게 사용하면] → [이런 결과가 나온다]
   - 시나리오 2: ...

## 6. 데이터 모델                                    ← 기존 유지
## 7. API 참조                                       ← 기존 유지
## 8. 환경 및 설정                                   ← 기존 유지

## 부록: 코드 레퍼런스 목록                          ← Source 테이블 통합
```

### 논문↔코드 스펙 최종 매핑 테이블

| 학술 논문 요소 | 코드 스펙 대응 | 섹션 |
|---------------|---------------|------|
| Abstract | 프로젝트 개요 + 핵심 가치 | §1 상단 |
| Introduction / Motivation | 배경, 문제 정의, 왜 이 프로젝트인가 | §1 |
| Related Work | 대안 기술/접근법 비교 | §1 하단 |
| Core Method / Algorithm | 핵심 설계 아이디어 + 실제 코드 발취 | §2 |
| Formulas | 코드 스니펫 (인라인 citation 포함) | §2, §4 |
| Implementation Details | 컴포넌트 상세 구현 디테일 | §4 |
| Experiments | 시나리오별 사용법 가이드 | §5 |
| Results | 각 시나리오의 기대 결과 | §5 |
| References / Citations | 인라인 코드 citation + 부록 레퍼런스 목록 | 본문 + 부록 |
| Supplementary Material | 데이터 모델, API 참조, 환경 설정 | §6-§8 |

## 리서치 결과 요약 (Research Findings)

- **현재 SDD 스펙 포맷 분석**: 목표/아키텍처/컴포넌트 상세/API/데이터 모델 등 14개 섹션. 레퍼런스/네비게이션 중심(70%), 서사적 설명(30%).
- **기존 "paper-like" 시도**: spec-summary에서 "paper-like subsections" 언급 있었으나, 실제 스펙에는 미적용.
- **Overview 섹션 개선**: 최근 discussion에서 컴포넌트에 Overview 섹션 추가하여 "왜 이 컴포넌트가 존재하는가" 서사 보강 시도 있었음.

## 토론 흐름 (Discussion Flow)

Round 1: [포맷 관계: 대체 vs 진화 vs 공존] → 기존 포맷 진화 결정
Round 2: [우선순위: 어떤 논문적 요소가 가장 급한가] → 배경/동기 > 핵심 아이디어 > 사용법 순서, 모두 부족
Round 3: [Citation 형태: 인라인 vs Source 테이블 vs 하이브리드] → 하이브리드 결정
Round 4: [코드 스니펫 깊이: pseudocode vs 실제 코드 vs 요약+링크] → 실제 코드 발취
Round 5: [Drift 관리: spec-update-done 강화 vs 수동 vs 타협] → spec-update-done 강화
Round 6: [기대 결과 매핑 + 사용법 가이드 깊이] → 기대 동작 서술 + 시나리오 기반 튜토리얼
Round 7: [구체적 스펙 구조 제시] → 순서/구성 수정 필요
Round 8: [구조 수정] → §5 기대 동작 + 부록A 사용 가이드를 "사용 가이드 & 기대 결과"로 통합
Round 9: [구조 확정 + 실행 방안] → 구조 확정, 실행 방안은 추후 논의
