# 토론 요약: Whitepaper 스타일 대형 스펙의 2-페이즈 생성 전략

**날짜**: 2026-03-13
**라운드 수**: 7
**참여 방식**: 구조화된 토론 (discussion skill)

## 핵심 논점 (Key Discussion Points)

1. **컨텍스트 윈도우 한계**: §1~§8 whitepaper 스타일 스펙을 한 번에 생성하면 LLM 컨텍스트 윈도우를 초과하거나 후반부 품질이 저하될 우려
2. **품질 일관성**: 한 패스로 대형 문서를 생성하면 뒷부분으로 갈수록 앞부분과의 일관성이 깨질 수 있음
3. **2-페이즈 접근법**: 건축의 골조→마감 비유로, 먼저 구조와 계획을 세운 후 내용을 채우는 방식 제안
4. **병렬 실행 가능성**: §1→§2→§3는 순차적 의존성이 있지만, §4~§8은 독립적으로 병렬 생성 가능
5. **스킬별 적용 범위**: 모든 스킬에 일률 적용이 아닌, 크기 기반으로 자동 판단하여 필요할 때만 2-페이즈 적용

### 스킬별 임계값 (초기 추정치)

**핵심 원칙**: LLM 단일 패스에서 품질 유지 가능한 마크다운 출력은 대략 **~300줄**. 이를 넘으면 후반부 품질이 눈에 띄게 저하됨.

| 스킬 | 측정 대상 | 1-페이즈 | 2-페이즈 | 근거 |
|------|----------|---------|---------|------|
| **spec-create** | 소스 파일 수 (테스트/설정 제외) | < 30개 | >= 30개 | 30개 이상이면 §4 Component Details만으로 상당한 분량 |
| **spec-rewrite** | 기존 스펙 줄 수 | < 300줄 | >= 300줄 | 직접 측정 가능, 출력 ≈ 입력 크기 |
| **spec-upgrade** | 기존 스펙 줄 수 | < 300줄 | >= 300줄 | rewrite와 동일 논리 |
| **spec-update-todo** | 추가할 기능 수 + 기존 스펙 크기 | < 3개 기능 AND 스펙 < 200줄 | 그 외 | 3개 이상이면 여러 섹션 동시 수정 |
| **spec-update-done** | 변경된 섹션 수 + 기존 스펙 크기 | < 3개 섹션 변경 AND 스펙 < 200줄 | 그 외 | 절반 이상 수정이면 사실상 rewrite |

> **참고**: 초기 추정치이며, spec-create에 먼저 적용 후 실험적으로 조정 예정.

## 토론 흐름 (Discussion Flow)

Round 1: 핵심 우려 확인 → 컨텍스트 윈도우 한계 + 품질 저하
Round 2: 해결 방향 탐색 → 사용자가 "다른 아이디어" 제안
Round 3: 아이디어 구체화 → **2-페이즈 (골조→채우기)** 방식 제안
Round 4: 골조 수준 결정 → 섹션별 미니 요약 (3-5줄)
Round 5: Phase 2 실행 방식 → 순차+병렬 조합, 컨텍스트는 골조 전체+코드베이스
Round 6: 적용 범위 및 관계 정리 → 크기 기반 복합 판단, 생성/저장 전략 독립
Round 7: 리뷰 프로세스 및 우선순위 → 자동 진행+후수정, spec-create부터 적용
