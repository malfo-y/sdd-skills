# 토론 요약: SDD 스펙의 사람을 위한 설명 부족 문제

**날짜**: 2026-03-09
**라운드 수**: 5
**참여 방식**: 구조화된 토론 (discussion skill)

## 핵심 논점 (Key Discussion Points)

1. **생성된 스펙 콘텐츠의 설명 부족**: 스킬 UX가 아니라, 스펙 문서 자체가 사람에게 코드와 기능을 충분히 설명하지 못하는 문제
2. **SDD_SPEC_REQUIREMENTS.md의 구조적 억제**: 요구사항 문서 자체가 "토큰 효율", "변경 우선", "코드 복사 방지" 원칙으로 인해 설명을 구조적으로 억제함
3. **참조/탐색 70% vs 설명 30% 불균형**: 현재 스펙 템플릿이 경로, 심볼, 계약, Change Recipe에 집중하고 동작 설명과 설계 의도가 부차적
4. **변경 범위**: SDD_SPEC_REQUIREMENTS.md 자체를 보강해야 하며, 스킬 템플릿만 바꾸는 것으로는 불충분
5. **메인 스펙과 컴포넌트 스펙 모두 보강 필요**: 컴포넌트 스펙뿐 아니라 메인 스펙의 Runtime Map도 서술형으로 보강

## 리서치 결과 요약 (Research Findings)

- **스킬 출력 분석**: 7개 스펙 관련 스킬 모두 참조/탐색 데이터(경로, 심볼, 계약)에 최적화. 서술형 설명은 Goal과 Responsibility 1-2문장에 한정
- **context-mode 충돌**: 500 word 제한과 artifact policy가 스펙 설명 확장과 구조적으로 충돌할 수 있음. 단, 이는 스킬 실행 시의 출력 제약이며 생성되는 스펙 파일 자체에는 영향 없음

## 토론 흐름 (Discussion Flow)

- Round 1: 문제 정의 → 스킬 UX가 아니라 **스펙 콘텐츠 자체**의 설명 부족 확인
- Round 2: 부족한 설명 유형 → **아키텍처/설계 의도 + 기능별 동작 설명** 두 가지

- Round 4: 섹션 설계 → Responsibility 유지 + **Overview 1개 섹션 추가** (동작 개요 + 설계 의도 통합)
- Round 5: 최종 범위 → 메인 스펙 Runtime Map도 보강, §6 예시 추가 포함
