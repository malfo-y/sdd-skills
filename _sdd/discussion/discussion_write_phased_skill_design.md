# 토론 요약: write-phased 스킬 설계

**날짜**: 2026-03-14
**라운드 수**: 7
**참여 방식**: 구조화된 토론 (discussion skill)

## 핵심 논점 (Key Discussion Points)
1. **스킬의 핵심 가치**: 긴 파일을 한 번에 쓰지 않고 skeleton → fill 2단계로 작성하는 전략
2. **적용 범위**: 마크다운 문서와 코드 파일 모두 지원, 각각에 맞는 skeleton 전략 별도 정의
3. **스킬 간 참조 불가**: 현재 Claude Code 스킬 시스템에서 스킬 간 import/참조 메커니즘이 없음 → 독립 스킬로 설계
4. **유연성 vs 구체성 균형**: 대부분의 세부 결정을 AI 판단에 위임하되, 핵심 규칙과 템플릿은 명시적으로 정의

## 리서치 결과 요약 (Research Findings)
- **spec-create/guide-create 2-phase 패턴**: Phase 1은 ~50-80줄 skeleton을 파일에 저장, Phase 2는 순차(의존 섹션) + 병렬(독립 섹션)로 채우기. `<!-- Phase 2에서 상세 작성 -->` 마커 사용.
- **스킬 간 참조 메커니즘**: 스킬은 독립적인 프롬프트 문서(SKILL.md)로, import/참조 메커니즘 없음. 스킬 간 소통은 생성된 파일을 통한 간접적 방식 + 사용자 주도 오케스트레이션.

## 토론 흐름 (Discussion Flow)
Round 1: 적용 범위 + 스킬 이름 → 문서+코드 병렬 지원, `write-phased`
Round 2: 호출 방식 → 사용자 질문으로 전환 (스킬 간 참조 가능 여부)
Round 3: 스킬 간 참조 불가 확인 → 독립 스킬로 확정
Round 4: 코드 skeleton + 호출 패턴 → AI 판단 위임, 자연어 트리거
Round 5: Phase 전환 + 파일 범위 → 자동 진행, AI 판단
Round 6: 스킬 밀도 + Phase 2 메커니즘 → 중간 밀도, Edit 교체
Round 7: 에지 케이스 → 트리거 기준(구조적 복잡도), 실패 처리(미정의), 적용 대상(새+추가)
