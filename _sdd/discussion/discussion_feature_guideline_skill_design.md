# 토론 요약: 기능 가이드라인 문서 작성 스킬 추가

**날짜**: 2026-03-12
**라운드 수**: 11
**참여 방식**: 구조화된 토론 (discussion skill)

## 핵심 논점 (Key Discussion Points)
1. 스킬 구조: 기존 스킬 확장보다 새 전용 스킬로 분리하는 편이 역할 경계가 명확하다.
2. 입력 기준: `_sdd/spec/`를 주 소스로 두고, 관련 코드를 함께 읽어 세부 구현 현실을 확인하는 하이브리드 방식이 적합하다.
3. 대상 범위: 첫 버전은 컴포넌트보다 기능 단위 가이드를 우선 지원하는 편이 SDD 흐름과 연결하기 쉽다.
4. 산출물 위치: 스펙 본문과 분리된 `_sdd/guides/` 전용 디렉토리가 자연스럽다.
5. 문서 구조: 설명, 규칙, 체크리스트, 예시를 모두 포함하는 계층형 구성이 실용적이다.
6. 작업 범위: 새 스킬은 가이드 문서만 생성하고, 스펙 수정이나 구현 반영은 기존 후속 스킬에 맡기는 편이 안전하다.

## 리서치 결과 요약 (Research Findings)
- `_sdd/spec/sdd_skills.md`: 현재 스킬 체계는 스펙, 초안, 구현, 리뷰 중심이며 가이드 전용 산출물 카테고리는 없음.
- `_sdd/env.md`: 이 저장소는 애플리케이션 런타임보다 스킬 프롬프트와 문서 자산을 관리하는 저장소에 가깝다.
- `.codex/skills/feature-draft/SKILL.md`: 새 기능 계획 초안과 구현 계획 생성에 초점이 있고, 별도 가이드 문서 생성 책임은 없음.
- `.codex/skills/spec-update-todo/SKILL.md`: 스펙 반영을 담당하며, 파생 가이드 문서 산출보다는 `_sdd/spec/` 수정이 핵심 역할이다.
- `.codex/skills/discussion/SKILL.md`: discussion 스킬은 의사결정 정리용이며, 새 기능을 설계하기 위한 토론 진입점으로 적합했다.

## 토론 흐름 (Discussion Flow)
Round 1: 구조 결정 -> 기존 스킬 확장 대신 새 전용 스킬 선호
Round 2: 입력 기준 확인 -> 스펙 중심 + 관련 코드 검증의 하이브리드 방식 선호
Round 3: 대상 범위 확인 -> 첫 버전은 기능 가이드 우선
Round 4: 출력 위치 확인 -> `_sdd/guides/` 전용 디렉토리 선호
Round 5: 문서 구조 확인 -> 설명, 규칙, 체크리스트, 예시를 모두 포함한 계층형 구성이 적합
Round 6: 최종 수렴 -> 가이드 문서만 생성하는 새 스킬로 정리하고, 부록에 스펙/컴포넌트 레퍼런스 추가
Round 7: 스킬 이름 결정 -> `guide-create`
Round 8: 파일명 규칙 결정 -> `guide_<slug>.md`
Round 9: 최소 필수 섹션 결정 -> 설명, 규칙, 체크리스트, 예시
Round 10: 컴포넌트 확장 전략 결정 -> 단일 스킬 내부 옵션 분기
Round 11: 부록 레퍼런스 범위 결정 -> 파일 경로 + 섹션/심볼 단위 포함
