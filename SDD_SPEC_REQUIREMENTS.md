# SDD 스펙 문서 요구사항

SDD에서 스펙 문서는 단순한 설명서가 아니라, 사람이든 LLM이든 코드베이스를 빠르게 이해하고 안전하게 변경하기 위한 탐색 가능한 지도여야 한다.

이 문서는 우리가 스펙 문서에 기대하는 핵심 목적, 필수 요구사항, 권장 구조, 품질 기준을 정리한다.

---

## 1. 스펙 문서를 만드는 이유

### 핵심 목적

1. **빠른 이해**
   - 사람이 이 코드베이스나 스펙이 무엇인지 빠르게 이해할 수 있어야 한다.

2. **탐색과 변경 지원**
   - 사람이나 LLM이 기능을 추가하거나 변경할 때, 어디에 무엇이 있는지 찾고 어디를 고쳐야 하는지 판단할 수 있어야 한다.

3. **결정과 제약의 보존**
   - 코드만으로 드러나지 않는 비직관적 구조, 제약, 트레이드오프, 불변 조건을 남겨야 한다.

4. **변경 위험 감소**
   - 어떤 변경이 어디까지 영향을 미치는지, 무엇을 함께 확인해야 하는지 드러내서 수정 비용과 회귀 위험을 줄여야 한다.

### 한 줄 정의

좋은 스펙은 코드의 복사본이 아니라, `의도 + 구조 + 계약 + 변경 지점 + 결정 이유`를 압축한 검색 가능한 레이어다.

---

## 2. 문서 철학

### 스펙은 무엇이어야 하는가

- 스펙은 백과사전식 코드 덤프가 아니라 **탐색형 문서**여야 한다.
- 스펙은 "이 저장소가 무엇을 하는가"와 "무엇을 어디서 바꿔야 하는가"를 우선적으로 알려줘야 한다.
- 메인 스펙은 상세 설명서보다 **진입점(entry point)** 역할을 해야 한다.
- 컴포넌트 스펙은 "이 컴포넌트가 무엇인가"보다 **"어디를 보면 되고 무엇을 깨면 안 되는가"** 를 더 강하게 드러내야 한다.

### 스펙과 코드의 관계

- **코드는 구현의 원본(source of truth)** 이다.
- **스펙은 이해와 변경을 위한 운영 인터페이스**다.
- 스펙은 코드를 대체하지 않는다.
- 스펙은 코드를 읽기 전에 방향을 잡아 주고, 코드를 수정할 때 탐색 시작점을 제공해야 한다.

### 모르는 것은 숨기지 않는다

- 확인되지 않은 내용은 추정으로 단정하지 않는다.
- 확신이 낮은 내용은 `Open Questions`로 분리한다.
- 틀린 확신보다 명시적 불확실성이 낫다.

### LLM이 스펙을 소비하는 맥락을 고려한다

- LLM은 컨텍스트 윈도우가 유한하다. 너무 긴 스펙은 오히려 핵심 정보를 밀어내서 해로울 수 있다.
- 메인 스펙은 LLM이 한 번에 읽고 전체 그림을 잡을 수 있는 분량이어야 한다. 상세는 컴포넌트 스펙으로 분리한다.
- LLM은 자연어 산문보다 **구조화된 포맷**(테이블, 리스트, `key: value`)을 더 정확하게 파싱한다.
- 경로와 심볼은 코드블록(`` ` ``)으로 감싸서 일반 텍스트와 구분해야 한다.

## 3. 필수 요구사항

| 요구사항 | 의미 | 문서에 드러나야 하는 형태 |
|---------|------|--------------------------|
| 빠른 첫 이해 | 처음 보는 사람이 5분 안에 전체 그림을 잡을 수 있어야 한다 | 프로젝트 요약, 시스템 경계, 핵심 흐름 |
| 검색 가능성 | 사람/LLM이 필요한 영역을 바로 찾을 수 있어야 한다 | 실제 경로, 컴포넌트 인덱스, 핵심 심볼 |
| 변경 지향성 | 기능 변경 시 어디부터 봐야 하는지 알려줘야 한다 | Change Guide, Change Recipes, 영향 범위 메모 |
| 책임 경계 명확화 | 각 컴포넌트가 무엇을 하고 하지 않는지 보여야 한다 | Responsibility, Owned Paths, Dependencies |
| 계약 가시화 | 외부/내부 인터페이스와 불변 조건이 보여야 한다 | Inputs/Outputs, API/Event 계약, Invariants |
| 위험 노출 | 무엇을 잘못 바꾸면 문제가 생기는지 보여야 한다 | Risks, Known Issues, 회귀 포인트 |
| 실행 가능성 | 로컬 실행/테스트/검증 시작점이 있어야 한다 | Environment, Setup/Test Commands |
| 결정 추적성 | 왜 이런 구조를 택했는지 남아야 한다 | `DECISION_LOG.md` 또는 간단한 rationale |
| 확장 가능성 | 대규모 코드베이스에서도 문서가 무너지지 않아야 한다 | main + component spec 분할 구조 |
| 최신성 유지 | 코드 변경 후 스펙도 같이 갱신되어야 한다 | spec-update 흐름과 연결 |

## 4. 기본적으로 포함해야 할 내용

### 메인 스펙(main spec)

메인 스펙은 프로젝트를 처음 보는 사람이 가장 먼저 여는 문서여야 하며, 다음 역할을 해야 한다.

1. 프로젝트가 해결하는 문제를 설명한다.
2. 저장소의 시스템 경계를 설명한다.
3. 주요 디렉토리/파일/컴포넌트의 지도를 제공한다.
4. 주요 런타임 흐름을 보여준다.
5. 기능 변경 시 시작 지점을 알려준다.
6. 공통 불변 조건과 중요한 제약을 드러낸다.
7. 남아 있는 불확실성을 분리해서 보여준다.

### 컴포넌트 스펙(component spec)

컴포넌트 스펙은 특정 책임 단위를 깊게 설명하는 문서이며, 다음 역할을 해야 한다.

1. 이 컴포넌트의 책임과 비책임을 설명한다.
2. 실제 소유 경로와 핵심 심볼을 연결한다.
3. 입력/출력/외부 계약을 정리한다.
4. 업스트림/다운스트림 의존성을 정리한다.
5. 특정 변경 유형별 시작 지점과 검증 포인트를 제공한다.
6. 관련 테스트, 로그, 메트릭, 디버깅 포인트를 보여준다.
7. 이 컴포넌트에서 특히 조심할 불변 조건과 위험을 기록한다.

### 별도 문서

- `DECISION_LOG.md`
  - 비직관적 결정, 트레이드오프, 구조적 제약을 남길 때만 사용한다.
- `_sdd/env.md`
  - 실행, 테스트, 환경 변수, 필수 서비스 등 환경 정보를 관리한다.

## 5. 권장 문서 구조

섹션은 **필수(MUST)**와 **선택(OPT)**으로 나뉜다. 필수 섹션만으로도 유용한 스펙이 되어야 하며, 선택 섹션은 프로젝트 규모와 복잡도에 따라 추가한다.

### 메인 스펙 권장 섹션

| 구분 | 섹션 | 설명 |
|------|------|------|
| **MUST** | `Goal` | 프로젝트 목적과 범위 |
| | ↳ `Project Snapshot` | 한 문단 요약 |
| | ↳ `Key Features` | 핵심 기능 목록 |
| | ↳ `Non-Goals` | 하지 않는 것 |
| **MUST** | `Architecture Overview` | 시스템 구조 |
| | ↳ `System Boundary` | 외부 경계와 연동 |
| | ↳ `Repository Map` | 디렉토리/파일 지도 |
| | ↳ `Runtime Map` | 주요 런타임 흐름 |
| OPT | ↳ `Technology Stack` | 기술 스택 정리 |
| OPT | ↳ `Cross-Cutting Invariants` | 전역 불변 조건 |
| **MUST** | `Component Details` | 컴포넌트 요약과 링크 |
| | ↳ `Component Index` | 이름, 책임, 경로, 스펙 링크 |
| OPT | `Environment & Dependencies` | 실행 환경 (별도 `env.md`로 분리 가능) |
| OPT | ↳ Setup / Test Commands | 셋업과 테스트 명령 |
| OPT | ↳ Configuration / Secrets | 설정과 시크릿 |
| OPT | `Identified Issues & Improvements` | 위험, 기술 부채, 미비 사항 |
| OPT | `Usage Examples` | 실행, 일반 작업, 변경 경로 예시 |
| OPT | ↳ `Common Change Paths` | 자주 하는 변경의 시작점 |
| **MUST** | `Open Questions` | 미확인/불확실 사항 |

> **가이드라인**: 작은 프로젝트(파일 50개 이하)라면 MUST 섹션만으로 충분하다. 컴포넌트가 5개 이상이거나 외부 연동이 복잡한 프로젝트부터 OPT 섹션을 채워 나간다.

### 컴포넌트 스펙 권장 섹션

| 구분 | 섹션 | 설명 |
|------|------|------|
| **MUST** | `Responsibility` | 이 컴포넌트가 하는 일과 하지 않는 일 |
| **MUST** | `Owned Paths` | 소유 파일/디렉토리 경로 |
| **MUST** | `Key Symbols / Entry Points` | 핵심 함수, 클래스, 진입점 |
| **MUST** | `Interfaces / Contracts` | 입력/출력, API, 이벤트 계약 |
| **MUST** | `Dependencies` | 업스트림/다운스트림 의존 |
| **MUST** | `Change Recipes` | 변경 유형별 시작점과 검증 포인트 |
| OPT | `Tests / Observability` | 테스트, 로그, 메트릭, 디버깅 포인트 |
| OPT | `Risks / Invariants` | 깨면 안 되는 조건, 주의 사항 |
| OPT | `Known Issues` | 알려진 문제와 우회 방법 |
| OPT | `Open Questions` | 미확인 사항 |

## 6. 구체적 예시: 좋은 스펙 vs 나쁜 스펙

원칙만으로는 실제 품질 수준을 판단하기 어렵다. 이 섹션은 핵심 영역별로 좋은 예시와 나쁜 예시를 비교한다.

### 6.1 Repository Map

**나쁜 예시** — 경로 없이 모호한 설명:

```
프로젝트는 서비스 레이어, 데이터 레이어, 프레젠테이션 레이어로 구성됩니다.
서비스 레이어에서 비즈니스 로직을 처리하고, 데이터 레이어에서 DB 접근을 합니다.
```

**좋은 예시** — 실제 경로와 역할을 연결:

```
src/
├── api/            # HTTP 엔드포인트 (Express 라우터)
│   ├── routes/     # 라우트 정의
│   └── middleware/  # 인증, 에러 핸들링
├── domain/         # 비즈니스 로직 (프레임워크 무관)
│   ├── order/      # 주문 생성, 상태 전이
│   └── payment/    # 결제 처리, 환불
├── infra/          # 외부 연동
│   ├── db/         # Prisma 스키마, 마이그레이션
│   └── external/   # PG사 API 클라이언트
└── config/         # 환경별 설정
```

### 6.2 Change Recipes

**나쁜 예시** — 무엇을 바꾸라는지 알 수 없음:

```
새 API를 추가하려면 관련 파일을 수정하세요.
```

**좋은 예시** — 단계, 경로, 검증 포인트를 구체적으로 제공:

```markdown
### 새 REST 엔드포인트 추가

1. `src/api/routes/`에 라우트 파일 추가
2. `src/domain/`에 비즈니스 로직 함수 작성
3. `src/api/routes/index.ts`에 라우트 등록
4. `tests/api/`에 통합 테스트 추가

검증: `npm test -- --grep "새 엔드포인트"` 통과
주의: `src/api/middleware/auth.ts`의 권한 체크 미들웨어 적용 필수
```

### 6.3 Interfaces / Contracts

**나쁜 예시** — 구현을 그대로 복사:

```typescript
// OrderService 클래스는 createOrder, cancelOrder, getOrderById,
// updateOrderStatus, listOrders, getOrderHistory, validateOrder,
// calculateTotal, applyDiscount, sendConfirmation 메서드를 가집니다.
// createOrder는 userId, items, shippingAddress를 받아서...
// (이하 200줄의 메서드별 설명)
```

**좋은 예시** — 외부에서 알아야 할 계약만 간결하게:

```
OrderService 핵심 계약:
- createOrder(userId, items, address) → Order
  - 불변: items가 비어있으면 거부
  - 불변: 재고 차감은 이 함수 안에서 트랜잭션으로 처리
  - 부작용: 주문 생성 후 `order.created` 이벤트 발행
- cancelOrder(orderId) → void
  - 불변: SHIPPED 이후 상태는 취소 불가
  - 부작용: 재고 복원, `order.cancelled` 이벤트 발행
```

### 6.4 Responsibility

**나쁜 예시** — 책임만 있고 비책임이 없음:

```
PaymentService는 결제를 처리합니다.
```

**좋은 예시** — 경계가 명확:

```
PaymentService
- 책임: PG사 API 호출, 결제 상태 추적, 환불 처리
- 비책임: 주문 상태 변경(OrderService), 알림 발송(NotificationService)
- 주의: 결제 상태와 주문 상태 동기화는 이벤트 기반. 직접 호출하지 않음.
```

### 6.5 Open Questions

**나쁜 예시** — 불확실성을 사실처럼 기술:

```
이 시스템은 초당 10,000건의 요청을 처리할 수 있습니다.
```

**좋은 예시** — 모르는 것을 명시:

```
## Open Questions
- 현재 부하 테스트 미실시. 실제 처리량 미확인.
- Redis 캐시 만료 정책이 비즈니스 요구사항과 맞는지 검증 필요.
- `external/pg-client.ts`의 타임아웃 설정 근거 불명 (현재 30초).
```

## 7. 대규모 코드베이스에서 스펙을 사용하는 방식

### 1단계: 빠른 온보딩

메인 스펙을 열어 프로젝트 목표, 시스템 경계, 주요 컴포넌트, 런타임 흐름을 먼저 파악한다.

목표는 "이 저장소가 무엇을 하는가"를 짧은 시간 안에 잡는 것이다.

### 2단계: 변경 시작점 찾기

기능을 바꾸려면 관련 컴포넌트 스펙에서 다음을 본다.

- 어떤 책임 단위가 관련되는가
- 실제로 어떤 경로를 봐야 하는가
- 어떤 계약을 깨면 안 되는가
- 어떤 테스트와 운영 포인트를 같이 확인해야 하는가

### 3단계: 영향 범위 판단

스펙은 "어디를 고쳐야 하는가"뿐 아니라, "어디까지 같이 봐야 하는가"를 알려줘야 한다.

특히 아래 영역은 영향 범위 판단이 중요하다.

- API 계약
- 이벤트 흐름
- 상태 전이
- 설정 변경
- 배치 작업
- 외부 서비스 연동

### 4단계: 리뷰와 검증

코드 리뷰는 라인 단위지만, 스펙은 시스템 단위 검증에 사용한다.

즉, 구현이 아래를 어기지 않았는지 본다.

- Goal
- Architecture Overview
- Component contract
- Invariants

### 5단계: 구현 후 동기화

아키텍처, 계약, 운영 방식, 변경 경로가 바뀌었으면 스펙도 같이 갱신해야 한다.

갱신하지 않으면 스펙은 금방 오래된 위키가 된다.

---

## 8. 스펙 갱신 기준

모든 코드 변경이 스펙 갱신을 요구하는 것은 아니다. 아래 기준으로 판단한다.

### 갱신이 필요한 변경 (MUST update)

| 변경 유형 | 예시 |
|----------|------|
| 컴포넌트 추가/삭제/이름 변경 | 새 서비스 모듈 추가, 기존 모듈 제거 |
| 외부 계약 변경 | API 엔드포인트 추가/변경, 이벤트 스키마 변경 |
| 아키텍처 구조 변경 | 디렉토리 구조 개편, 새 외부 서비스 연동 추가 |
| 불변 조건 변경 | 상태 전이 규칙 변경, 유효성 검증 조건 변경 |
| 런타임 흐름 변경 | 동기→비동기 전환, 새 큐/워커 추가 |
| 환경/의존성 변경 | 새 환경변수, DB 마이그레이션, 주요 라이브러리 교체 |

### 갱신이 불필요한 변경 (NO update)

| 변경 유형 | 예시 |
|----------|------|
| 내부 구현 리팩토링 | 함수 내부 로직 개선, 변수명 변경 |
| 버그 수정 (계약 불변) | 기존 계약대로 동작하지 않던 것을 수정 |
| 테스트 추가 | 기존 기능에 대한 테스트 보강 |
| 성능 최적화 (인터페이스 불변) | 쿼리 최적화, 캐시 추가 (외부 동작 동일) |
| 코드 스타일/포맷팅 | lint 수정, import 정리 |

### 갱신 고려가 필요한 변경 (CONSIDER)

| 변경 유형 | 판단 기준 |
|----------|----------|
| 새 내부 유틸/헬퍼 추가 | 다른 컴포넌트에서 공유하면 갱신 |
| 에러 핸들링 정책 변경 | 외부로 노출되는 에러 형태가 바뀌면 갱신 |
| 설정값 변경 | 동작 방식이 달라지면 갱신 |
| 의존 라이브러리 업데이트 | 메이저 버전 변경이면 갱신 |

## 9. 좋은 스펙의 작성 원칙

### 1. 경로 우선(path-first)

모호한 설명보다 실제 파일/디렉토리 경로, 엔트리포인트, 심볼을 우선한다.

### 2. 변경 우선(change-first)

"어떻게 동작하는가" 설명도 중요하지만, "어디를 바꾸면 되는가"가 더 중요하다.

### 3. 지도 우선(index-first)

메인 스펙은 모든 것을 길게 설명하기보다, 어디에 무엇이 있는지 빠르게 찾게 해 줘야 한다.

### 4. 분할 가능성(split-friendly)

문서가 커지면 `main + component specs` 구조로 나눠야 한다.

### 5. 결정 추적(decision traceability)

비직관적인 선택은 `DECISION_LOG.md`에 짧게라도 남겨야 한다.

### 6. 불확실성 명시(explicit unknowns)

모르는 것을 숨기지 말고 `Open Questions`에 적어야 한다.

### 7. 토큰 효율(token-efficient)

LLM이 소비하는 맥락에서, 같은 정보를 더 적은 토큰으로 전달할 수 있다면 그렇게 한다. 산문 나열보다 테이블과 리스트가 낫고, 반복 설명보다 참조 링크가 낫다.

## 10. 피해야 할 안티패턴

각 안티패턴에 대해 왜 문제인지, 어떻게 고치는지를 함께 설명한다.

| 안티패턴 | 왜 문제인가 | 대안 |
|---------|------------|------|
| 코드를 거의 그대로 복사한 문서 | 코드가 바뀌면 즉시 불일치. 읽어도 새로운 정보가 없음 | 계약과 의도만 남기고, 구현 상세는 코드에 맡긴다 |
| 장황한 구현 설명부터 시작하는 문서 | 프로젝트 목적을 파악하기 전에 포기하게 됨 | Goal/Project Snapshot을 최상단에 배치 |
| 실제 경로나 심볼이 없는 문서 | 검색 시작점이 없어서 결국 코드를 뒤져야 함 | 모든 컴포넌트에 Owned Paths, Key Symbols 명시 |
| 변경 포인트가 없는 문서 | "알겠는데 어디를 고치지?"에 답할 수 없음 | Change Recipes 섹션 추가 |
| 모든 프로젝트에 보안/성능 상세를 무조건 붙이는 문서 | 관련 없는 정보가 핵심을 가림. 토큰 낭비 | 프로젝트에 실제로 해당하는 항목만 포함 |
| 불확실한 내용을 사실처럼 쓰는 문서 | 잘못된 정보를 신뢰하고 행동하면 사고로 이어짐 | Open Questions로 분리 |
| 메인 문서 하나에 모든 것을 몰아넣은 문서 | LLM 컨텍스트를 초과하고, 사람도 길을 잃음 | main + component spec 분할 |
| 코드 변경 후 갱신되지 않는 문서 | 시간이 지나면 오히려 혼란을 가중 | 8장의 갱신 기준에 따라 동기화 |

## 11. 품질 체크리스트

아래 질문에 대부분 "예"라고 답할 수 있어야 한다.

### 이해

- 이 문서만 보고 신규 기여자가 5분 안에 프로젝트 목적을 이해할 수 있는가?
- 시스템 경계가 명확한가?
- 주요 흐름이 짧고 명확하게 드러나는가?

### 탐색

- 주요 컴포넌트마다 실제 경로나 심볼이 연결되어 있는가?
- 어떤 기능을 바꿀 때 어디부터 봐야 하는지 찾을 수 있는가?
- 관련 테스트와 디버깅 시작점을 찾을 수 있는가?

### 안정성

- 중요한 계약과 불변 조건이 보이는가?
- 영향 범위가 큰 변경 지점이 드러나는가?
- Known Issues / Risks가 숨겨지지 않았는가?

### 유지보수성

- 문서 구조가 스케일 가능한가?
- 불확실성이 `Open Questions`에 분리되어 있는가?
- 코드 변경 후 갱신할 위치가 명확한가?

### LLM 효율성

- 메인 스펙이 LLM 컨텍스트 한 번에 들어가는 분량인가?
- 구조화된 포맷(테이블, 리스트)을 사용하고 있는가?
- 경로와 심볼이 코드블록으로 구분되어 있는가?
- 같은 정보를 중복 기술하고 있지 않은가?

## 12. 결론

SDD 스펙 문서는 "코드 설명"이 아니라, **코드베이스를 이해하고 안전하게 바꾸기 위한 운영 인터페이스**여야 한다.

좋은 스펙은 다음을 동시에 만족해야 한다.

1. 사람이 빠르게 이해할 수 있다.
2. 사람과 LLM이 변경 지점을 찾을 수 있다.
3. 비직관적 결정과 제약을 보존한다.
4. 변경의 영향 범위와 위험을 줄인다.
5. 코드와 함께 계속 갱신된다.
6. LLM의 컨텍스트 제약 안에서 효율적으로 소비될 수 있다.
