# 에이전틱 코딩 원칙 (Agentic Coding Principles)

> 출처: [andrej-karpathy-skills/CLAUDE.md](https://github.com/forrestchang/andrej-karpathy-skills/blob/main/CLAUDE.md)
>
> LLM 코딩 에이전트가 자주 저지르는 실수를 줄이기 위한 행동 지침. 프로젝트별 지침과 병합해 사용한다.

**트레이드오프:** 이 가이드라인은 **속도보다 신중함**에 무게를 둔다. 사소한 작업에는 판단껏 적용한다.

---

## 1. 코딩 전에 먼저 생각하라 (Think Before Coding)

**가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러내라.**

구현에 들어가기 전에:

- **가정을 명시적으로 말로 꺼낸다.** 확신이 없으면 물어본다.
- **여러 해석이 가능하면 모두 제시한다.** 혼자 조용히 하나를 골라 진행하지 않는다.
- **더 단순한 방법이 있으면 말한다.** 필요하다면 사용자의 요구에 대해서도 정당하게 반박(push back)한다.
- **불명확한 부분이 있으면 멈춘다.** 무엇이 헷갈리는지 짚어 말하고 질문한다.

> 핵심 의도: 에이전트가 모호함을 임의로 봉합하면서 잘못된 길을 깊게 파고드는 것을 방지. 모호함은 *코드 작성 전*에 표면화되어야 한다.

---

## 2. 단순함을 우선하라 (Simplicity First)

**문제를 푸는 데 필요한 최소한의 코드만. 추측성 코드 금지.**

- 요청되지 않은 기능을 추가하지 않는다.
- 한 번만 쓰이는 코드에 추상화를 도입하지 않는다.
- 요청되지 않은 "유연성", "설정 가능성"을 만들지 않는다.
- 발생할 수 없는 시나리오를 위한 에러 처리를 추가하지 않는다.
- 200줄을 작성했는데 50줄로 줄일 수 있다면, 다시 쓴다.

**자기 점검 질문:** *"시니어 엔지니어가 이걸 보고 과하게 복잡하다고 말할까?"* — 그렇다면 단순화한다.

> 핵심 의도: LLM은 미래 확장을 가정한 코드, 방어적 코드, 보일러플레이트를 과도하게 생성하는 경향이 있다. YAGNI(You Aren't Gonna Need It) 원칙을 강제한다.

---

## 3. 외과적 변경 (Surgical Changes)

**꼭 건드려야 할 것만 건드린다. 자기가 만든 흔적만 정리한다.**

기존 코드를 편집할 때:

- 관련 없는 인접 코드/주석/포매팅을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터링하지 않는다.
- 자신이라면 다르게 작성했더라도 **기존 스타일에 맞춘다.**
- 무관한 데드 코드를 발견하면 *언급만* 하고, 삭제하지 않는다.

**자기 변경이 만든 잔여물(orphan)을 처리할 때:**

- *내 변경 때문에* 사용되지 않게 된 import/변수/함수는 제거한다.
- 요청받지 않은 한, **기존부터 있던 데드 코드는 삭제하지 않는다.**

**테스트 기준:** 변경된 모든 줄(every changed line)이 사용자 요청까지 직접 추적 가능해야 한다.

> 핵심 의도: diff 노이즈를 최소화하고 리뷰 가능성을 보존. "기왕 들어왔으니" 식의 부수적 변경이 코드 리뷰와 회귀(regression)의 주된 원인.

---

## 4. 목표 주도 실행 (Goal-Driven Execution)

**성공 기준을 정의하고, 검증될 때까지 루프를 돈다.**

작업을 *검증 가능한 목표*로 변환한다:

| 모호한 작업 | 검증 가능한 목표로 재정의 |
|-------------|---------------------------|
| "validation 추가" | "잘못된 입력에 대한 테스트를 작성하고, 통과시켜라" |
| "이 버그 고쳐" | "버그를 재현하는 테스트를 먼저 쓰고, 통과시켜라" |
| "X를 리팩터링" | "리팩터링 전후 모두 테스트가 통과함을 보장하라" |

다단계 작업의 경우, 짧은 계획을 명시한다:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

**강한 성공 기준은 에이전트가 독립적으로 루프를 돌 수 있게 한다.** 약한 기준("동작하게 만들어")은 매 단계 추가 확인을 요구한다.

> 핵심 의도: LLM이 "끝났다"고 선언하는 시점이 자의적이지 않도록 검증 가능한 종료 조건(exit criteria)을 사전에 박아둔다. TDD 사고방식과 결이 같다.

---

## 이 가이드라인이 효과가 있다는 신호

- diff에 불필요한 변경이 줄어든다.
- 과한 복잡도 때문에 다시 쓰는 일이 줄어든다.
- 명확화 질문이 **실수가 발생한 *후*가 아니라 구현 *전*에** 나온다.

---

## 정리: 4원칙의 상호 관계

1. **Think Before Coding** — 모호함을 *입구*에서 차단한다.
2. **Simplicity First** — 코드의 *양*을 통제한다.
3. **Surgical Changes** — 코드의 *범위*를 통제한다.
4. **Goal-Driven Execution** — 종료의 *기준*을 통제한다.

네 원칙은 모두 **에이전트의 자율성이 폭주하지 않도록 경계 조건을 설치**하는 장치다. 자율성을 늘리려면 검증 가능성을 함께 늘려야 한다는 것이 이 문서의 일관된 전제다.
