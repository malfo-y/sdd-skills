# Spec-Driven Development(SDD)

AI Agent 시대의 개발 방식 전환

## TL;DR (1페이지 요약)

> **SDD(Spec-Driven Development)** 는 “문서 작성”이 아니라, AI 에이전트가 흔들리지 않도록
**명세(Spec)를 진실의 공급원(Source of Truth)으로 두고** 구현/검증을 그 아래에 정렬하는 개발 방식이다.
>

### AI 코딩에서 자주 터지는 3가지 실패 패턴

1. **해석의 표류(Interpretation Drift)**: 같은 요구인데 구현 결정이 매번 달라짐
2. **컨텍스트 윈도우 한계**: 코드베이스가 커질수록 맥락을 잃고 중복/무시/우회가 증가
3. **환각(Hallucination)**: 존재하지 않는 API/라이브러리를 “있는 것처럼” 만들어냄

### SDD가 제공하는 3가지 제어 장치

- **의도 선언(What)과 구현(How)의 분리**
- **표류 감지(Drift detection)**: Spec↔︎코드가 어긋나는 순간을 빠르게 찾는 장치
- **가드레일(Guardrails)**: 금지 패턴/원칙/제약(=헌법)을 Spec에 박아 AI의 자유도를 제한

## 0. 오프닝: 이 문서의 목표

> ❝ SDD를 ’이론’으로 아는 게 아니라 **AI agent와 같이 일하기 위한 실전 도구**로 이해한다 ❞
>

### 왜 지금 “의도 중심”인가

소프트웨어 공학은 “구현 중심”에서 “의도 중심”으로 이동하고 있습니다. AI 에이전트가 코딩의 주도권을 잡기 시작한 지금, 중요한 것은 코드 생성 속도가 아니라 **생성 결과물이 시스템의 전체 설계/비즈니스 의도와 일치하는지**입니다.

이 문서는 SDD를 단순한 문서화가 아니라 **실행 가능한 계약(Executable Spec)** 으로 보고, 그 계약을 기준으로 구현과 검증을 정렬해 “AI를 안전하게 빠르게 쓰는 방법”을 다룹니다.

## 1. 문제 제기: AI agent 개발, 왜 이렇게 불안한가?

### AI agent 코딩을 할 때 우리가 이미 겪고 있는 증상

- 같은 요구를 줘도 결과가 매번 다름 (AI의 랜덤성)
- “이 정도는 알아서 하겠지” → 결과는 제각각
- 프롬프트가 사실상 **구두 계약**
- 사람/모델이 바뀌면 맥락이 증발

### AI agent의 실패 패턴 3종 (핵심)

### 1) 해석의 표류(Interpretation Drift)

- **현상**: 팀/에이전트/날짜가 바뀌면 구현 선택이 바뀜 (AI의 랜덤성)
- **예시**: 같은 ‘인증’ 요구인데 agent A는 JWT, agent B는 세션 쿠키로 구현 → 통합 시 충돌
- **SDD 관점 처방**: “결정”을 Spec에 박아 **결정론적으로 재사용**

### 2) 컨텍스트 윈도우 한계

- **현상**: 레포가 커질수록 기존 패턴/유틸/규칙을 놓치고 중복 기능을 생성
- **예시**: 이미 있는 validation 유틸을 무시하고 새 유효성 검사기를 또 만듦
- **SDD 관점 처방**: Spec이 “필수 맥락의 압축본”이 되어, 에이전트의 입력을 안정화

### 3) 환각(Hallucination)과 거짓 기능

- **현상**: 존재하지 않는 API/라이브러리를 만들어서 “되는 것처럼” 코드를 작성
- **SDD 관점 처방**: I/O, 계약, 제약을 Spec에 명시해 **검증 가능하게** 만들기

#### (용어 정리) I/O, 계약(Contract), 제약(Constraints), “검증 가능”

- **I/O**: 특정 기능(특히 라이브러리 함수/모듈)이 **무엇을 입력으로 받고(Input)** **무엇을 출력으로 내는지(Output)**를 뜻합니다.
  - 포함해야 하는 정보: 입력/출력 형태(타입, 필수/옵션, 범위), 성공/실패 결과(리턴/에러/예외), 부작용(side effects: I/O, 상태 변경) 유무
  - I/O가 비어 있으면 에이전트가 빈칸을 추측으로 채우기 쉬워져 환각이 늘어납니다.

- **계약(Contract)**: 구현과 사용자가 서로 지켜야 할 **검증 가능한 약속**입니다. ("그럴듯함"이 아니라 "맞/틀"을 판정할 수 있는 문장)
  - **Preconditions(전제조건)**: 호출/사용 전에 참이어야 하는 조건(입력 유효성, 전제 상태)
  - **Postconditions(사후조건)**: 성공/실패 각각에 대해 호출 후 반드시 성립해야 하는 보장(반환값 성질, 상태 변화)
  - **Invariants(불변조건)**: 시스템/도메인이 항상 유지해야 하는 규칙(유일성, 보존 법칙 등)
  - 포인트: 계약을 명시하면 리뷰/테스트의 질문이 “코드가 그럴듯한가?”가 아니라 “계약을 지키는가?”로 바뀝니다.

- **제약(Constraints)**: "무엇을 구현할지"가 아니라, 구현이 **절대 넘으면 안 되는 경계/가드레일**입니다.
  - 예) 성능/복잡도 한계, 보안/컴플라이언스, 호환성(기존 동작 유지), 의존성(새 라이브러리 추가 금지), 결정론/재현성 요구, 외부 I/O 금지 등
  - 포인트: 제약은 에이전트가 편의상 선택하는 지름길(새 라이브러리 도입, 존재하지 않는 API 호출 등)을 차단합니다.

- **검증 가능(Verifiable)**: Spec의 문장이 **테스트/정적검사/리뷰 체크리스트로 기계적으로 변환**될 수 있는 상태입니다.
  - 좋은 신호: “언제/무엇을 넣으면/무슨 결과가 나와야 한다”가 명확하고, 반례(실패 모드)까지 정의되어 있음
  - 나쁜 신호: “적당히/빠르게/안전하게/잘 동작하게”처럼 판정 기준이 없는 표현

### “프롬프트=구두 계약”의 한계

바이브 코딩의 핵심 문제는 “대화 기록”이 팀의 계약이 되기 어렵다는 점입니다. 프롬프트는 애매하고, 사람/모델/날짜가 바뀌면 해석이 달라지며, 코드베이스가 커지면 맥락이 빠르게 증발합니다.

- **비구조화 입력**(자연어 프롬프트)은 팀 합의가 축적되지 않음 → 결과가 “구두 계약”에 의존
- **결정이 재사용되지 않음** → 다음 에이전트/다음 날/다른 사람이 같은 문제를 다시 풀고, 답이 달라짐
- **레포 규모가 커질수록** 에이전트가 맥락을 잃음 → 중복 기능/규칙 위반/패턴 무시가 증가
- **환각**이 “코드”로 들어오면, 추적 가능한 계약/근거가 부족해 회복이 늦어짐

SDD는 이 문제를 “더 좋은 프롬프트”로 푸는 대신, **결정과 제약을 Spec으로 외부화**해서 누구든(사람/에이전트) 같은 입력을 받으면 같은 결론에 도달하도록 만드는 접근입니다.

## 2. SDD란 무엇인가 (엔지니어 관점)

### 한 줄 정의

> **SDD = 코드보다 먼저, 사람과 AI가 동시에 읽는 Spec을 만들고, 그 Spec을 계약으로 운영한다**
>

여기서 Spec은:

- 문서(설명)만이 아니라
- **시스템의 계약(Contract)** 이자 **변경의 기준선(Baseline)**

### SDD의 핵심 철학 3가지

1. **Spec is the Source of Truth** (코드는 결과물)
2. **중요한 결정은 Spec에 남긴다** (누가/언제 구현해도 같은 결론)
3. **검증 기준도 Spec에 둔다** (리뷰/테스트/계약 검증)

### “Spec이 진실”이라는 말의 실무적 의미

- 이제부터 리뷰/테스트의 질문이 “코드가 그럴듯한가?”가 아니라 **“이 구현이 Spec(계약)을 지키는가?”**로 이동합니다.
- Spec을 SoT로 두면, 에이전트/개발자가 바뀌어도 **같은 입력(계약/제약)**에서 같은 결론(구현/테스트/검증)으로 수렴합니다.
- 운영 원칙 하나만 기억하면 됩니다: **AI가 만든 Spec도 사람 승인 없이는 확정하지 않는다.** (환각이 ’계약’으로 굳는 순간이 가장 위험)

### 왜 AI agent 개발에서는 SDD 도입이 더 쉬워졌나

SDD는 원래 “좋은데 무겁다”는 인상이 강했습니다. 그런데 AI agent가 구현 생산성을 폭발적으로 올리면서, 역설적으로 **Spec의 비용은 낮아지고(작성/정리), Spec의 효용은 커져서(가드레일/검증)** 도입 장벽이 크게 낮아졌습니다.

#### 기존(사람 중심 개발)에서 도입이 어려웠던 이유

- **작성 비용 대비 체감 ROI가 낮음**: 사람이 직접 코드를 쓰는 환경에서는 “일단 구현하고 나중에 문서화”로 흐르기 쉬워 Spec이 부가물로 밀립니다.
- **동기화 문제(Spec drift)**: Spec과 코드가 쉽게 어긋나는데, 이를 잡아내는 장치가 없으면 Spec이 금방 신뢰를 잃습니다.
- **프로세스 저항**: “문서를 더 쓰자”로 들리면 팀은 즉시 부담으로 인식하고, 도입이 ‘프로세스 개편’으로 느껴져 반발이 큽니다.
- **검증이 약하면 허무해짐**: Spec이 테스트/체크리스트로 내려오지 않으면, 결국 리뷰는 다시 “그럴듯함”으로 회귀합니다.

#### AI agent 시대에 장벽이 낮아진 이유(그리고 SDD가 바로 돈값을 하는 지점)

- **구현 속도가 빨라질수록 ‘틀리게’ 구현할 확률도 같이 커짐**: AI는 빠르지만 의도 추측/환각에 취약합니다. 그래서 Spec(계약/제약)이 없으면 재작업이 더 비싸집니다.
- **Spec 초안 작성 비용이 급감**: AI가 Spec의 초안을 빠르게 만들고, 사람은 Clarify로 빈칸/의사결정 포인트를 채우는 방식이 가능합니다.
- **함수/모듈 단위에 특히 잘 맞음**: 라이브러리 개발은 I/O와 계약(전제/사후/불변)을 문장과 테스트로 내리기 쉬워, Spec→Unit test→Implement 루프가 짧게 돌아갑니다.
- **‘검증 가능’이 도구로 강화됨**: 계약 테스트/스키마 검증/정적 분석/CI 체크를 붙이면 Spec↔︎코드 정합성 감지가 자동화되어 drift가 줄어듭니다.

또한 AI는 “코드를 대신 쓰는 것”뿐 아니라, 아래 같은 **Spec 중심 작업을 대신/자동화**해 **SDD의 비용을 낮추고 품질을 올리는 역할**을 할 수 있습니다.

- **Spec 초안/템플릿 채우기**: Goal/Non-goals/Definitions/컴포넌트 골격을 빠르게 생성
- **Clarify 질문 리스트업**: 구현 전에 결정이 필요한 질문(선택지/리스크 포함) 자동 생성
- **Acceptance Criteria(=테스트 후보) 생성**: 인터페이스별 정상/경계/오류 케이스를 Given/When/Then으로 제안
- **누락/충돌/모호함 탐지**: I/O·에러모델·제약의 빠진 부분, 섹션 간 불일치(용어/정책/금지사항) 경고
- **Spec→Plan/Tasks 변환 보조**: 구현 계획/작업 분해(우선순위, 영향 범위, 검증 포인트) 초안 생성
- **Spec↔︎Code 드리프트 리뷰**: PR diff/테스트를 보고 “스펙에 있는데 구현 없음 / 구현 있는데 스펙 없음”을 리포트로 정리

#### 결론: 도입의 핵심은 “문서를 더 쓰자”가 아니라 “검증 기준을 Spec으로 바꾸자”

- SDD를 도입할 때 가장 큰 오해는 “Spec 작성량을 늘리는 것”입니다.
- 실제로는 **PR/리뷰의 Definition of Done을 Spec(계약/제약/검증) 기준으로 고정**하는 것이 출발점이고, 그 다음에 프로세스/도구는 점진적으로 올리면 됩니다.

> (도입 방법/로드맵은 뒤의 운영 프로세스와 부록의 ‘도입’ 섹션에서 더 구체적으로 다룹니다.)

### 전통적 개발 (Code-first) vs SDD (의도/사양-first)

| 구분 | 전통적 개발(Agile/Code-First) | SDD(Spec-Driven Development) |
| --- | --- | --- |
| 진실의 공급원 | 구현 코드(Implementation) | 실행 가능한 사양(Executable Spec) |
| 인간의 역할 | 코드 작성/로직 구현 | 시스템 설계/의도 정의(Orchestrator) |
| AI의 역할 | 자동완성/스니펫 생성 | 사양 기반 구현 + 검증(테스트/계약) |
| 변경 관리 | 코드 수정 + 문서 사후 업데이트 | **사양 수정 → 코드 재생성/재정렬** |
| 주요 아티팩트 | Commit, Unit Test | Spec, Plan, Data Model |

### (보너스) SDD vs TDD / BDD (AI 관점)

| 관점 | SDD | TDD | BDD |
| --- | --- | --- | --- |
| 출발점 | 명세(의도/제약/계약) | 테스트(정확성) | 시나리오(행동) |
| 주요 질문 | “무엇을 보장해야 하나?” | “이 함수가 맞나?” | “사용자는 어떻게 쓰나?” |
| AI 친화성 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| 대규모 변경 | 강함(계약 기반) | 약함 | 중간 |

> 결론: **SDD는 TDD/BDD를 대체하지 않는다.SDD가 위에 있고, TDD/BDD는 아래에 온다.** (레이어링)

## 3. SDD의 ‘제어면(Control Plane)’: 메커니즘 3개

핸드아웃 관점에서 SDD를 “문서화 기법”이 아니라 **아키텍처 제어 장치**로 이해하면 빠릅니다.
즉, 아키텍처를 ’권고사항’이 아니라 **사양을 통해 강제되는 규칙(실행 가능한 의도)** 으로 다룹니다.

### Control Plane을 한 문장으로

SDD는 **의도(What) → 구현(How)**로 내려가는 통로를 Spec으로 통제해서, 표류/환각/팀 규칙 위반을 “개별 실수”가 아니라 **시스템 차원의 검증 문제**로 바꾸는 접근입니다.

### 1) 선언적 의도(What)와 구현(How)의 분리

- Spec은 what (Goal/Non-goals/Contract/Invariants)
- 구현은 how (언어/프레임워크/최적화)

이 분리는 **비즈니스 로직의 영속성**을 높이고 (기술이 바뀌어도 의도 유지), 레거시 현대화(예: Node.js → Go) 같은 큰 변화에서 리스크를 낮춥니다.

### 2) 표류 감지(Drift detection)

- Spec과 코드가 어긋나는 순간(=표류)을 빠르게 감지
- 예: 계약 테스트, 스키마 검증, payload 검사, spec diff 리뷰

아키텍처/계약을 **런타임 불변량(invariants)** 처럼 취급하고, 운영 중에도 지속적으로 강제합니다.
“한 번 리뷰하고 끝”이 아니라, CI에서 spec↔︎code 정합성을 계속 확인하는 형태로 강화할 수 있습니다.

### 3) 가드레일(Guardrails)

- AI가 ’그럴듯한데 팀 표준과 다른 코드’를 만들지 못하게
- 헌법(Constitution)으로 **금지 패턴/원칙/제약**을 명시

### AI 에이전트와의 공생(역할 분담)

- **AI는 패턴 인식/구현 자동화에 강하지만, 의도 추측에는 한계**가 있습니다.
- Spec/Constitution은 AI가 “마음대로” 해석하지 못하도록 하는 **최상위 가이드라인/제약(Guardrail)** 입니다.
- 중요한 운영 원칙: **AI가 생성한 사양도 반드시 인간이 승인**해야 합니다(환각을 사양으로 확정하는 리스크 방지).

## 4. 운영 프로세스: 팀이 내일부터 따라할 루프

핵심은 “초기에 완벽한 사양”이 아니라, **모호성을 앞단에서 정리하고 (clarify) 작은 루프를 반복**하는 것.

### Clarify 10~20분이 ’가장 싼 보험’인 이유

Specify 단계의 초안은 보통 “그럴듯하지만 빈칸”이 남습니다. Clarify는 그 빈칸을 질문으로 빠르게 드러내고(무엇을 결정해야 하는지), 결정 로그로 고정해서 **뒤에서 폭발할 재작업을 앞에서 상쇄**합니다.

| 단계 | 목적 | 산출물(예) | 포인트 |
| --- | --- | --- | --- |
| Constitution | 불변 원칙/금지 패턴 고정 | constitution.md | 일관성 가드레일 |
| Specify | 의도/목표/범위 정의 | feature spec | 무엇을 만들지 합의 |
| Clarify | 빈칸 메우기 | Q&A / 결정 로그 | **10~20분 투자로 재작업 감소** |
| Plan | 기술 계획/접점 정리 | implementation plan | 종속성/리스크 |
| Tasks | 작은 단위로 분해 | task list | 1~3파일 단위 권장 |
| Implement | 구현/검증/리뷰 | code + tests | 리뷰 기준은 Spec |

### 단계별로 “무엇을 적어야 하는가”

- **Constitution(헌법)**: 프로젝트 불변 규칙/금지 패턴을 “명문화”
  - 예: “에러 처리는 전역 미들웨어에서만 수행, 라우터에서 try/catch 금지” 같은 식으로 **구체적인 금지 규칙**을 적습니다.
- **Specify(사양)**: 기술스택보다 **사용자 여정/성공 기준/비즈니스 제약**에 집중
- **Clarify(명확화)**: 10~20분의 질문/결정 로그가 **재작업 리스크를 크게 줄임**
  - “빈칸이 어디인지”, “의사결정이 필요한 포인트가 무엇인지”를 강제로 드러냅니다.
- **Plan(기술 계획)**: 데이터 모델, API 계약, 통합 지점, 라이브러리/제약 정리
  - 특히 브라운필드에서는 AI에게 **기존 코드베이스를 먼저 ‘연구’** 하게 해서 중복 구현을 줄입니다.
- **Tasks(작업 분해)**: 각 작업은 가급적 **1~3개 파일 범위**로 제한(LLM의 대규모 변경 품질 저하 방지)
- **Implement(구현)**: 인간 리뷰는 구문(syntax)보다 **사양 준수/아키텍처 적합성** 중심으로 전환

## 5. 실무에서 바로 쓰는 최소 Spec 템플릿

### “AI에게 주기 충분한” 최소 Spec

이 섹션의 템플릿은 **스킬 문서 기준**으로 정리합니다.

- **메인 스펙(SoT)**: `/spec-create`가 만드는 `_sdd/spec/<project>.md` 또는 `_sdd/spec/main.md`
- **업데이트 입력(요구사항 묶음)**: `user_spec.md`(직접 작성) / `user_draft.md`(`/spec-draft`) → `/spec-update-todo`가 메인 스펙에 반영

아래는 메인 스펙을 “AI에게 주기 충분할 정도로만” 남기는 **최소 골격**입니다. (우리가 실제로 쓰는 구획: Description / Component Details / Acceptance Criteria)

```markdown
# <Project Name>

## 0. Description

### Goal
- 프로젝트/모듈의 목적(무엇을 해결?)

### Non-Goals
- 이번 범위에서 하지 않는 것(명시적으로 제외)

### Definitions
- 핵심 용어/약어 정의

### Success Criteria (선택)
- “성공”을 어떻게 판정하는가(예: 정확도/성능/운영 지표)

### Constraints (선택)
- 성능/보안/호환성/의존성/외부 I/O 금지 등

## 1. Component Details

### <Component Name>

| Aspect | Description |
|---|---|
| **Purpose** | 이 컴포넌트가 하는 일 |
| **Input** | 입력 타입/형태/전제조건 |
| **Output** | 출력 타입/형태/성공 조건 |
| **Dependencies** | 내부 모듈/외부 의존성(있다면) |

**Architecture Details (선택)**
- 구현 접근(큰 방향)
- Key classes/functions

**How to Use (선택)**
- 사용 예시(짧게)

**Known Issues (선택)**
- 한계/리스크/추후 개선

## 2. Acceptance Criteria

- [ ] (정상 케이스) 입력 X를 주면 출력 Y가 나온다
- [ ] (에러 케이스) 조건 Z에서는 에러/예외가 난다(에러 모델 포함)
- [ ] (제약) 성능/보안/의존성 제약을 위반하지 않는다
```

### 이 템플릿이 “AI에게 충분”한 이유

1. **Description에 Goal/Non-Goals/Constraints가 함께** 있어, AI가 “범위 밖”을 마음대로 확장하기 어렵습니다.
2. **Component Details 표**로 I/O·의존성·책임이 고정되어, 존재하지 않는 API/모듈을 지어내는 환각을 줄입니다.
3. **Acceptance Criteria**가 테스트/리뷰의 최소 단위가 되어, 구현 결과를 “맞/틀”로 빠르게 검증할 수 있습니다.

### “최소”가 중요한 이유(Over-spec 방지)

- Spec은 모든 설명을 담는 문서가 아니라 **결정/계약/제약의 압축본**이어야 합니다.
- 너무 자세한 Spec(Over-spec)은 컨텍스트를 낭비하고 속도를 늦춥니다. 대신 Goal/Non-goals로 범위를 박고, Inputs/Outputs+Invariants로 **검증 가능한 경계**를 만들면 AI가 마음대로 추측하기 어렵습니다.

## 5A. Markdown Spec이 기본(대부분 충분)

- **Markdown Spec = Source of Truth(의도/범위/결정/제약)**
- **OpenAPI/TypeSpec = Boundary Contract(인터페이스 계약을 기계가 검증/생성)**

대부분의 초기/단일 팀 개발은 **Markdown Spec만으로 충분**합니다.
OpenAPI/TypeSpec은 팀/시스템 경계를 넘는 인터페이스를 **자동으로 검증·생성·호환성 강제**해야 할 때 도입합니다.
 도입 신호/체크리스트/점진적 승격 루프는 문서 하단 **부록 A**를 참고하세요.

## 5B. 과잉 스펙 방지(Over-spec) & 레벨링 규칙

AI가 스펙을 과하게 디테일하게 만들지 않게 하려면, “잘 쓰기”보다 **규칙으로 상한선을 거는 것**이 효과적입니다.

### 1) Spec Budget(상한선) — 길이/깊이를 제한

- L0/L1(아래 레벨 정의 참고)는 합쳐서 **2~3페이지(또는 N줄)**를 넘기지 않기
- 본문에 들어갈 예시는 **핵심 플로우 1개 + 에러 예시 1개**까지만
- Edge case는 “나열”하되, 처리 알고리즘까지 쓰지 않기(필요하면 L2로)

### 2) MUST는 계약, 설명은 분리

- **MUST/SHALL**: 반드시 지켜야 하는 계약(검증 대상)
- **SHOULD/MAY**: 권장/선택
- “왜(맥락/근거)”는 본문을 늘리는 주범이므로 **Rationale/Decision log(L2)**로 분리

### 3) 스펙에 쓰면 안 되는 것(금지)

- 특정 파일 구조/클래스명/함수명/알고리즘 등 **구현 지시(How)**
- 아직 합의되지 않은 최적화/성능 튜닝 디테일
- 팀 표준과 무관한 장황한 설명(튜토리얼/교과서식 정의)

### 4) 레벨링(계층) — “짧게 유지”하면서도 필요한 디테일은 담는 방법

- **L0: One-pager(의도/범위)**
  - Goal / Non-goals / 성공 기준 / 핵심 제약(불변조건)
- **L1: Contract Spec(검증 가능한 경계)**
  - Inputs/Outputs, 오류 모델, 상태/플로우, Invariants
  - (필요 시) 여기서 OpenAPI/TypeSpec으로 승격
- **L2: Rationale / Decision log(Why/대안/리스크)**
  - 트레이드오프, 대안 비교, 운영 고려, 근거 링크
- **L3: Implementation Plan(How/작업 단위)**
  - 모듈 경계, 마이그레이션/릴리즈, 작업 분해(1~3파일 단위 권장)

### 5) 리뷰 체크리스트(과잉 스펙 감지)

- [ ] 본문(L0/L1)에 구현(How) 디테일이 섞였는가? → L3로 이동
- [ ] “당연한 설명”이 길게 반복되는가? → 삭제/요약
- [ ] 변경 가능성이 큰 디테일(내부 구조/함수명)이 계약처럼 고정되어 있는가? → 제거
- [ ] MUST(검증 대상)와 설명(참고)이 섞여 문서가 비대해졌는가? → 분리

### 6) AI에게 시킬 때의 운영 규칙(폭주 방지)

- 먼저 Clarify 질문 5개를 만들고, 답이 없으면 **Assumptions로 표시**
- 초안은 **L0/L1 (+사용자와 합의된 decision log)까지만 작성하고 멈추기** (필요한 디테일은 “추가 필요”로만 표시)
- “구현 지시 금지”를 Constitution/프롬프트에 명시

## 6. 우리 팀에서의 “작은 시작” 로드맵

### 1주차: 기능 1개에만 적용

- 신규 기능(또는 작은 개선) 1개를 고르고, Spec을 먼저 작성
- 구현은 AI에게 맡기되, 리뷰 기준은 Spec으로 고정

### 2주차: Spec 리뷰 루프 만들기

- “구현 논의 금지”가 아니라, **먼저 계약을 합의**
- AI를 리뷰 보조자로 활용: AI가 스펙을 읽고 결정이 필요한 질문 리스트를 만듦
- 질문(Clarify)을 템플릿으로 고정: 빈칸/모호함/의사결정 포인트

### 도입을 ’프로세스 변경’이 아니라 ’리뷰 기준 변경’으로 시작

가장 현실적인 첫걸음은 “Spec을 더 쓰자”가 아니라 **PR의 Definition of Done을 Spec으로 고정**하는 겁니다. 구현은 AI가 빨리 만들게 하되, 사람은 (1) Spec 정합성, (2) Constitution 위반 여부, (3) 검증(테스트/계약) 유무를 중심으로 본다는 원칙부터 팀에 심습니다.

### 안티패턴(워터폴 회귀 방지)

- ❌ 모든 것을 미리 다 쓰기(과도한 정교함)
- ✅ Feature 단위로 짧게 반복(수시간~수일)
- ✅ 상세 근거/도구 비교는 **부록**으로 보내 본문을 가볍게

### 단계별 도입 로드맵(조직 관점)

1. **Spec-Aware**: 코딩 방식은 유지하되, **Markdown Spec으로 계약/결정을 먼저 합의**하는 문화 만들기
2. **Spec-Led**: **AI(에이전트) + 체크리스트/산출물 규칙 + 리뷰 루프**로 **Specify→Plan→Tasks** 단계를 프로세스에 “강제”
   - (예) `/spec-review`로 질문/리스크 리포트 생성 → (인간 승인/결정) → `/spec-update-done`으로 Spec(SoT) 반영
3. **Spec-Synced/Automated**: CI/CD에 **spec↔︎code 정합성 검증**을 넣고, spec 변경 시 SDK/테스트 자동 갱신까지 연결

### 도입 시 필수 고려 사항

- **인간-AI 협업 모델**: AI=구현 효율, 인간=설계/최종 검증(감독관)
- **도구 선택**: 핵심은 “도구”가 아니라 **운영 규칙(템플릿/산출물 경로/리뷰 게이트)**이며, 필요하면 Spec Kit/Kiro 같은 도구나 사양 언어(TypeSpec/OpenAPI 등)로 자동화/검증을 보강
- **문화/교육**: “사양 작성이 곧 코딩”이라는 인식 + 리뷰 초점을 구문→의도/아키텍처로 이동

## 7. Q&A 치트시트 (10분)

발표자가 바로 답할 수 있도록 “한 줄 답 + 보조 포인트”로 구성합니다.

1. **Q. 이거 워터폴 아닌가요?**

- A. 전체를 미리 설계하는 워터폴이 아니라, **Feature 단위로 짧게 반복하는 루프**입니다.
- 보조: Clarify 10~20분 투자로 재작업을 줄이고, 변경 시 Spec 수정→코드 재정렬이 빠름

1. **Q. 문서(사양) 유지비용이 너무 큰데요?**

- A. “모든 설명”이 아니라 **결정/계약/제약만** 유지합니다.
- 보조: 상세한 배경/근거는 부록/링크로 분리

1. **Q. TDD/BDD는 버리나요?**

- A. 아니요. SDD는 위 레이어(의도/계약), TDD/BDD는 아래 레이어(검증)입니다. 자세한 내용은 [부록 G](#부록-g-tddbdd-실전-정리-sdd-맥락)를 참고하세요.

1. **Q. AI에게 어떻게 시키는 게 제일 효과적이죠?**

- A. “구두 프롬프트” 대신 **Spec + 금지/원칙(헌법) + 작업 단위(Tasks)** 로 지시하세요.

1. **Q. 성공을 어떻게 측정하죠?**

- A. 배포 후 결함/재작업/리뷰 시간, 온보딩 시간 같은 **운영 지표**로 봅니다.

1. **Q. 너무 디테일하게 Spec을 쓰면 오히려 느려지지 않나요?**

- A. 맞습니다. 그래서 SDD는 **본질적 의도/제약/계약**에 집중하고, 나머지는 링크/부록으로 보냅니다.

1. **Q. Spec에 ’왜(맥락)’가 빠지면, AI가 엉뚱한 결론을 내지 않나요?**

- A. 위험합니다. 그래서 Clarify/Decision log로 **의사결정 근거(Why)** 를 최소한으로 남기고,
“Tribal knowledge”를 조금이라도 문서화합니다.

1. **Q. AI가 만든 Spec을 그대로 믿어도 되나요?**

- A. 안 됩니다. **모든 Spec은 인간 승인**이 필요하고, 특히 외부 API/라이브러리 등은 검증 가능한 근거를 남겨야 합니다.

## 부록 A. (참고) TypeSpec vs OpenAPI (읽을거리)

> 본문에는 넣지 않고, 관심 있는 사람이 더 읽을 수 있게 부록으로 둡니다.
>

| 특징 | OpenAPI (YAML/JSON) | TypeSpec |
| --- | --- | --- |
| 코드 가독성 | 장황/반복이 많음 | 비교적 간결, TS 개발자에게 친숙 |
| 재사용/추상화 | $ref 중심 | 제네릭/템플릿/믹스인 등 추상화 강함 |
| 유효성 검사 | 런타임 스키마 검증 중심 | 컴파일 타임 타입 체크 지원(도구 기반) |
| 관리 단위 | 파일 단위 관리가 어려움 | 네임스페이스/패키지 시스템 지원 |
| 생산성 | 수작업 작성 시 오류 발생 높음 | IDE 인텔리센스/자동 완성 강함 |
| 출력 | 그 자체가 표준 | OpenAPI 등으로 변환하는 “상위 언어” 성격 |

실무적 가이드:

- 이미 OpenAPI가 조직 표준이면: **OpenAPI 유지 + Spec의 계약/검증 문화를 먼저**
- API가 커지고 반복이 많아지면: TypeSpec 같은 상위 언어를 **도입 검토**

### OpenAPI/TypeSpec을 쓰면 좋은 신호(체크리스트)

아래 중 **2개 이상**이면 OpenAPI/TypeSpec 도입을 강하게 고려합니다.

1. **팀/레포 경계를 넘는 API** (다른 팀, 다른 서비스, 외부 고객, 모바일 앱 등)
2. **동시 개발로 충돌이 자주 나는 인터페이스** (FE/BE, 서비스 간)
3. **SDK/클라이언트/Mock/문서 포털 생성**이 필요함
4. 요청/응답/이벤트에 대해 **스키마 검증을 CI/런타임에 걸고 싶음**
5. **버전 호환성/마이그레이션**을 계획적으로 운영해야 함
6. “말로는 합의했는데 구현이 다름”이 반복됨 → **계약 강제력 부족**

### Markdown만으로 충분한 신호(체크리스트)

- 단일 팀/단일 코드베이스, 소비자가 제한적
- 아직 빠르게 바뀌는 영역(탐색/프로토타입)이라 계약을 고정하기 이르다
- 스키마 자동 생성/검증보다 **의도·범위·결정(Goal/Non-goals)** 합의가 더 중요하다

### 점진적 형식화(Progressive Formalization) 권장 루프

1) 먼저 Markdown으로 **경계 계약의 ‘표면’**만 잡습니다(필드/오류/예시 1개).
2) 그 경계가 안정화되면, **그 부분만** OpenAPI/TypeSpec으로 승격합니다.
3) CI에서 최소 1개(예: 스키마 검증/계약 테스트)라도 걸어 “계약=자동 강제”를 만듭니다.

## 부록 C. SDD의 전략적 가치와 ROI(상세)

> 아래 수치는 deep_research에 포함된 업계 사례/리서치 인용을 그대로 옮긴 것이며, 팀/도메인에 따라 편차가 있을 수 있습니다.
>

### 정량적 성과(예시)

- **배포 시간 단축**: 명확한 사양 기반 AI 코딩으로 배포 시간을 최대 **50% 단축**
- **결함률 감소**: 사양-코드 자동 검증으로 배포 후 결함 **약 40% 감소**
- **API 개발 가속**: 계약 주도(Contract-Driven) 접근으로 사이클 타임 **75% 단축**(사례)
- **마이그레이션 효율**: 사양 기반 자동화로 작업의 **80% 자동화**, 전체 시간 **50% 절감**(사례)

### 정성적 가치(장기 이점)

1. **지식의 자산화 & 온보딩 가속**: 시스템 지식이 개인이 아니라 Spec에 누적
2. **레거시 현대화 용이성**: 비즈니스 규칙이 의도(What)로 분리되어 기술 교체가 쉬움
3. **협업 모호성 제거**: FE/BE가 공유 사양(OpenAPI/TypeSpec 등)으로 통합 충돌 감소
4. **보안/규정 준수 강화**: 보안 요구/규제 규칙을 Constitution/Spec에서 선제 강제

## 부록 D. 리스크 관리(상세): ’워터폴의 귀환’인가?

### SDD와 워터폴의 차이(요지)

- **시간 규모**: 워터폴은 수개월~수년, SDD는 기능 단위로 **수시간~수일 루프**
- **유연성**: 워터폴 사양은 변경이 어렵지만, SDD는 Spec 수정 후 **코드 재생성/재정렬**이 가능
- **피드백 루프**: 구현 이전에 Spec을 검토/명확화(Clarify)하고 지속적으로 정합성 검증

### 운영상의 제약/안티 패턴(요지)

1. **사양 유지관리 비용(유지관리 세금)**: Spec-first 문화가 없으면 Spec이 무용지물
2. **컨텍스트(Why) 누락**: “무엇(What)”만 남고 “왜(Why)”가 빠지면 의도에서 벗어날 수 있음
3. **과도한 정교함(Over-specification)**: 컨텍스트 윈도우 낭비 + 속도 저하
4. **AI 생성 사양의 맹신**: 환각을 Spec으로 ’확정’해버리는 최악의 실패

권장 완화책(실무)

- “모든 설명”이 아니라 **계약/제약/결정**만 Spec으로 유지
- Decision log로 최소한의 **Why**를 남기기
- 큰 기능은 쪼개고(Feature/phase), 작업은 1~3파일 단위로

## 부록 E. SDD 도입 체크리스트(상세)

### 단계별 도입(요약)

1. Spec-Aware → 2) Spec-Led → 3) Spec-Synced/Automated

### 도입 시 체크리스트(권장)

- [ ]  Constitution(원칙/금지 패턴) 합의가 문서로 존재한다
- [ ]  Spec 템플릿이 팀 표준으로 정해져 있다(Goal/Non-goals/Contract/Invariants 포함)
- [ ]  Clarify 질문 리스트가 있고, 결정 로그가 남는다
- [ ]  Plan 단계에서 통합 지점/데이터 모델/API 계약이 명시된다
- [ ]  CI에서 spec↔︎code 정합성(계약 테스트/스키마 검증 등)을 최소 1개라도 수행한다
- [ ]  리뷰 기준이 “구문”보다 “Spec 준수/아키텍처 적합성”에 맞춰져 있다

## 부록 F. 미래 전망: Self-healing / Closed-loop 아키텍처

- 런타임에서 Spec과 다른 동작이 감지되면, AI 에이전트가 자동 수정/경고하는 **자가 치유(Self-healing)** 가 가능해질 전망
- 시스템 성능/운영 데이터가 Spec으로 피드백되어 설계가 진화하는 **폐쇄 루프(Closed-loop)** 형태가 강화될 전망
- 엔지니어 역할은 “키보드 치는 사람”에서 **지능형 시스템의 설계자/감독관**으로 재정의

## 부록 G. TDD/BDD 실전 정리 (SDD 맥락)

### 왜 이 부록이 필요한가

- SDD는 “의도/계약(What)”을 고정하는 상위 레이어입니다.
- TDD/BDD는 그 계약을 “검증 가능한 실행 형태”로 내리는 하위 레이어입니다.
- 즉, **SDD가 방향을 정하고, TDD/BDD가 품질을 증명**합니다.

### TDD(Test-Driven Development) 상세

한 줄 정의:

- **TDD는 테스트를 먼저 작성하고(실패 확인), 최소 구현으로 통과시키고, 리팩터링으로 품질을 높이는 반복 루프**입니다.

핵심 루프(Red-Green-Refactor):

1. **RED**: 수용 기준(AC)을 테스트 케이스로 번역하고, 실패를 확인한다
2. **GREEN**: 테스트를 통과시키는 최소 코드만 작성한다
3. **REFACTOR**: 중복/가독성/구조를 개선하고 전체 테스트가 계속 통과하는지 확인한다

실무 적용 체크리스트(우리 방식):

- 수용 기준 1개당 최소 1개 테스트를 연결한다
- 테스트는 가능하면 `test_<unit>_<scenario>_<expected>` 형식으로 명명한다
- 테스트 본문은 Arrange-Act-Assert(AAA) 구조를 유지한다
- 기준별 테스트 통과 후, 마지막에 전체 테스트를 다시 실행해 회귀를 확인한다
- 리팩터링은 동작을 바꾸지 않는 범위에서만 수행한다

TDD가 특히 효과적인 구간:

- 도메인 규칙/에러 모델이 명확한 서비스 로직
- API 요청-응답 계약(정상/경계/오류)이 분명한 엔드포인트
- 회귀 리스크가 큰 리팩터링/모듈 분리 작업

### BDD(Behavior-Driven Development) 간략 정리

한 줄 정의:

- **BDD는 사용자/비즈니스 관점의 행동 시나리오를 Given-When-Then으로 명세해 팀의 공통 이해를 맞추는 방식**입니다.

SDD와의 연결:

- SDD의 상위 요구를 “사용자 행동 언어”로 구체화할 때 BDD가 유용합니다.
- 작성된 시나리오는 이후 TDD 테스트 케이스의 입력 재료가 됩니다.

짧은 예시:

- Given: 기존 회원 이메일이 존재한다
- When: 같은 이메일로 회원가입을 요청한다
- Then: 시스템은 409 응답을 반환한다

### 정리: 레이어링 원칙

- **SDD**: 무엇을 보장할지(의도/계약) 결정
- **BDD**: 어떤 사용자 행동을 만족해야 하는지 시나리오화
- **TDD**: 그 시나리오/계약을 테스트와 코드로 검증
